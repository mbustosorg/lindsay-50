## Why

Today `heart-message-manager/main.py:api_messages()` strips the inbound Twilio webhook down to `From` + `Body` and discards everything in `NumMedia*` / `MediaUrl*` / `MediaContentType*`. The two effect classes that *can* render attached media — `PngDisplay` and `VideoDisplay` in `lib_shared/patterns/` — exist as code, but neither the message wire nor the rotation surfaces them as first-class media paths: the rotation's registry at `lib_shared/config/effects_settings.json` (post-PR-53, the JSON-driven registry replaces the prior `_DEFAULT_EFFECTS_LIST_FULL` constant) carries them as listed-but-disabled entries that consume files from `<repo>/design/{pngs,videos}/`, which is operator-supplied, not message-driven. The result: when a user sends an SMS/MMS with a photo, the picture silently disappears, the message text scrolls on a Flame/NightSky background, and the operator has to physically copy the file onto the Pi and reboot for it to ever appear.

This change wires MMS media end-to-end: capture Twilio's `MediaUrl0..N` + `MediaContentType0..N`, immediately copy each attachment to OUR S3 (Twilio's retention is short and reading past it returns 410 — irrelevant to us once the bytes are in our bucket), carry the `(type, s3_key)` list on the existing `Message` wire, and teach the `EffectsCoordinator` to use that media as the background for the message's display cycle — falling back to the rotation when there's no media. The S3-key-on-the-wire pattern is what makes the URL durable across Twilio's deletion window: the Pi and browser fetch from OUR S3 via an authenticated Flask proxy, never from Twilio.

While doing the image work we'll rename `PngDisplay` → `ImageDisplay` and add JPEG/GIF/WebP support via PIL — the expand is local (PIL handles all four), the rename drops the misleading "Png" suffix the moment the class grows beyond PNG. The rename is a JSON edit against the post-PR-53 canonical at `lib_shared/config/effects_settings.json` (one entry replaced, one deprecated-alias entry added) plus a class/module rename; the existing `lib_shared/effects_loader.py:make_effect_class` resolves both names to the renamed class via dynamic import.

**Builds on PR-53.** PR-53 (#52) replaced the hardcoded effects registry with `lib_shared/config/effects_settings.json` (canonical) + `config_overrides/effects_settings.json` (gitignored operator override, REPLACE semantics) + a single `lib_shared/effects_loader.py` that dynamically imports each effect's class on demand. PR-53 also added a wire-strip behavior on the Pi: when the operator override is active, the wire's `effects_settings` block is dropped before `update_from_dict` runs (other top-level blocks — `text_settings`, `filters`, `senders`, `sign`, `timezone` — still come from the wire). This change extends that JSON-driven registry with a new `ImageDisplay` effect (and a deprecated `PngDisplay` alias), and uses the loader's dynamic import for the renamed class. The `MediaCycler` is per-message (built from `Message.media` at the `out → in` transition) and is unaffected by PR-53's wire-strip — see Design D11.

## What Changes

**Twilio ingestion (`heart-message-manager/main.py:_process_inbound_message`)**

- Detect `NumMedia` > 0 from `request.form`. **Respond to Twilio with 200 + TwiML immediately** — do not block the response on media downloads. The Twilio webhook response window is 15 s; we never want to spend it waiting on S3 puts (Design D13).
- A background `threading.Thread` (or `ThreadPoolExecutor` over the media list, downloaded in parallel) handles media processing. For each `MediaContentType{i}` / `MediaUrl{i}` pair: **download the bytes via Twilio Basic Auth** (`AccountSid:AuthToken` from the same `TWILIO_AUTH_TOKEN` env var used for webhook signature validation) and **write them to OUR S3 under `media/images/{YYYY-MM}/...` (for `image/*`) or `media/videos/{YYYY-MM}/...` (for `video/*`)**. GIF is `image/gif` and lands under `media/images/`. Failed downloads or S3 puts drop that item from the list (with WARNING); the text still publishes.
- Once the background thread completes (all uploads succeeded or failed), the `Message` (text + completed `media: list[{type, s3_key}]` list) is persisted to S3 + SQLite, and the `MessageEnvelope` is published over MQTT exactly once. There is no window in which a `MessageEnvelope` exists with media URLs that aren't yet durable in S3 — that's the R6 fix.
- A `MessageSid`-keyed dedupe guard prevents double-processing if Twilio retries the webhook while the background thread is still running.
- The wire tuple stored on `Message.media[*].url` is the **S3 key in our bucket** (e.g. `media/images/2025-12/media-2025-12-07T15-30-00Z.jpg`) — **not** the original Twilio `MediaUrl*`. After the bytes are copied, the Twilio URL is discarded; the wire never references Twilio's storage again. The Pi and browser fetch from OUR S3 via the Flask 302 endpoint.
- Match the existing message-logging fire-and-forget tone: try/except around the S3 write, log a WARNING and continue with an empty media list if S3 fails. The text always ships even if the media upload breaks — the operator sees a graceful text-only background, not a missing message.
- **Empty body with media is accepted; empty body + no media still 204.** Today `_process_inbound_message` returns 204 when `Body` is empty, which would drop a media-only MMS. Change the gate: accept the message iff `body` is non-empty OR `NumMedia > 0`; persist (text + media list) via S3 and MQTT. A media-only message flows through the coordinator with `body=""` and the existing `scroller.set_text("", display.width)` path in the `out → in` transition handles the blank-text case — the `MediaCycler` is the background, the scroller renders nothing, hold and rotation proceed normally.

**Message model (`lib_shared/models.py:Message`)**

- Add `media: list[dict]` (default `[]`) to `Message`. Each entry is `{"type": <mime>, "url": <s3-key>}`. Existing 4-field messages round-trip unchanged: `from_dict` defaults the field to `[]` when absent; `to_dict` always emits it.
- The wire shape grows by exactly one optional top-level array. Per-message filtering (`MessageView`) carries the media list verbatim — no filter behavior changes; suppressed messages just disappear, media and all.

**S3 storage (`heart-message-manager/s3.py`)**

- New `_MEDIA_KEY_TEMPLATE_IMAGES = "media/images/{year}-{month}/{basename}-{datetime}{ext}"` and `_MEDIA_KEY_TEMPLATE_VIDEOS = "media/videos/{year}-{month}/{basename}-{datetime}{ext}"`. Per the issue: a `/videos` and `/images` directory under the bucket, with the same `YYYY-MM` folder strategy. The `media/` prefix under each keeps `media/images/` and `media/videos/` distinct from any future `media/audio/`.
- Two new helpers: `log_media(content_type: str, source_url: str) -> str | None` (download → write to S3 → returns the S3 key on success or `None` on failure) and `proxy_media(s3_key: str) -> bytes | None` (for the Flask proxy endpoint below). `log_messages` S3 key templates and `load_messages_from_s3` stay exactly as-is — the existing message-body files are a different S3 prefix from media.
- New `MEDIA_KEY_PREFIXES = ("media/images/", "media/videos/")` for the existing `load_messages_from_s3`-style rebuild path, so a `rebuild_from_s3` start that scans all `messages/` and `media/` prefixes doesn't trip over media files (it does NOT scan media files; only the message-json files).

**Image fetch path (Flask 302 → signed S3 URL)**

- The Pi (Raspberry Pi) does not have AWS credentials in `settings.toml`. The browser preview runs entirely client-side and has no S3 SDK. Both need to fetch media bytes for `MediaCycler`'s `ImageDisplay` / `VideoDisplay` constructors. The Flask process has boto3 + AWS creds already, so it serves as the auth boundary:
  - `GET /api/media/<path:s3_key>` — `api_login_required`. Internally calls `s3.signed_media_url(s3_key)`, which returns `boto3.client("s3").generate_presigned_url("get_object", Params={"Bucket": bucket, "Key": key}, ExpiresIn=3600)`. Flask returns **`302 Found`** with `Location: <signed-s3-url>`. The Pi and browser follow the redirect; bytes come directly from S3 — never through Flask.
  - 1-hour TTL on the signed URL. The TTL is invisible to the wire: the Pi holds a logical `s3_key`, not a signed URL. Each fetch is `Pi → Flask → 302 → S3 → Pi`, with Flask re-generating a fresh signed URL on every request. The Pi holding a message for >1h simply re-fetches (gets a fresh URL); no client-side refresh logic.
  - The S3 key stored in the `Message.media[*].url` field is the *logical* key (`media/images/2025-12/img-…jpg`). Callers construct the Flask endpoint URL via `cfg.API_BASE_URL + "/api/media/" + s3_key`. Flask handles the S3 URL generation on-demand.
- Streaming bytes through Flask was considered and rejected: every media fetch would consume Flask bandwidth + CPU for the body transfer, doubling egress cost on Heroku for no auth benefit (the signed URL already carries the S3-side auth in its query string). The 302 redirect keeps the auth boundary on Flask (Pi/browser never see AWS creds) while moving the bytes through S3 directly.
- Pre-baking the signed URL into the wire (`Message.media[*].url` IS a signed URL) was considered and rejected: TTL becomes the operator's problem (Pi holds the URL forever, eventually 403s); Flask's admin UI can't safely re-display media from old messages without re-signing; every consumer of the wire has to know "this URL has a TTL, refresh before display."

**Image / Video display (`lib_shared/patterns/image_display.py`, `lib_shared/patterns/video_display.py` — internal to `MediaCycler`, removed from the effects registry)**

- Rename `lib_shared/patterns/png_display.py` → `lib_shared/patterns/image_display.py`, class `PngDisplay` → `ImageDisplay`. The class already supports palette-based rendering through the indexed Bitmap path, so a JPEG/GIF/WebP just changes the load: replace `Image.open(path).convert("RGBA")` with `Image.open(path).convert("RGB")` (drop the alpha, with conditional alpha-mask preservation — see Design D9), and replace the `glob("*.png")` with `glob("*")` filtered to a `_IMAGE_EXTS = (".png", ".jpg", ".jpeg", ".gif", ".webp")` tuple. GIF is a special case: PIL's `.convert("RGB")` discards the animation; keep it single-frame unless a follow-up change asks for animated GIF.
- **Remove `ImageDisplay` and `VideoDisplay` from the canonical effects registry at `lib_shared/config/effects_settings.json`.** The post-PR-53 canonical currently carries them as listed-but-disabled entries (consuming operator-supplied content from `<repo>/design/{pngs,videos}/`); both entries come out. The rotation is now strictly the 5 non-media effects (Hyperspace, Honeycomb, Flame, Fireworks, NightSky). The renamed `ImageDisplay` and the existing `VideoDisplay` are inner renderers consumed by `MediaCycler` via direct Python imports (`from lib_shared.patterns.image_display import ImageDisplay`, etc. — see Design D7) — they are not rotation effects anymore. Operator overrides (`config_overrides/effects_settings.json`) that still carry `PngDisplay`, `ImageDisplay`, or `VideoDisplay` entries hit the loader's existing unknown-name WARNING branch and are silently skipped. The curated-content idle-state use case (operator drops files into `design/pngs/`, the sign cycles them as background) is replaced by MMS — send yourself an MMS to get an image onto the sign.
- `lib_shared/effects_loader.py:make_effect_class` requires no change — the renamed classes are no longer in the registry, so the loader never sees them; the existing unknown-name WARNING branch handles any operator-override carryover.
- `VideoDisplay` stays as-is (already supports MP4/MOV/AVI/MKV/WebM/GIF). The constructor's `path` argument keeps working; per-message `MediaCycler` instantiates it with the downloaded media path.

**Per-message background override (`lib_shared/effects_coordinator.py`)**

- The coordinator already has a `get_display_message()` that returns the body to scroll. Add a sibling `get_display_media(message_id) -> list[dict] | None` (or extend the message pull to also return the media) that returns the media list of the *currently-displayed* message (the one whose body is on the scroller). When non-empty, the coordinator's `tick()` constructs an ad-hoc Effect for *this one display cycle*: a `MediaCycler` that holds a list of media items, picks one at a time, instantiates either an `ImageDisplay` (image/* mime) or a `VideoDisplay` (video/* mime) for the picked item, and renders through it.
  - `MediaCycler.media_hold_seconds = max(10.0, media_duration_or_image_interval)`. The image case uses the existing `ImageDisplay` defaults (currently 8 s); the video case uses the source video duration (read once via OpenCV's `cv2.CAP_PROP_FRAME_COUNT` / `cv2.CAP_PROP_FPS` at construction).
  - Multiple media items: cycle randomly each `media_hold_seconds`. **Decision: cut off at `hold_seconds`** rather than extend it. Issue flagged "either extend hold_seconds, or just cut off the display" — extending makes the timing unpredictable (a 10-minute hold vs the configured 15s hold); cutting off matches the existing rotation timeout behavior. The user sees each media item for at least its natural duration (or 10 s, whichever is longer); if the message had 8 photos and only 3 fit in `hold_seconds=15s`, the user sees 3. The remaining 5 are persisted in S3 (per the storage contract) and the operator can view the full set via the admin UI.
  - **Bad-codec items are removed from the cycler's list (Design D12).** If `VideoDisplay` (or `ImageDisplay`) raises a decode failure on the current item — codec mismatch, corrupt bytes, OpenCV can't read the frame, etc. — the cycler logs a WARNING, drops that item, and advances to the next. If the list becomes empty after removal, the cycler yields back to `self.effects[self.idx]` (the rotation) on the next fade. This keeps the panel alive across bad attachments instead of blacking out for the message's whole `hold_seconds` window.
  - When the message has fewer than 2 media items, the cycler just renders the one item for the natural duration (or 10 s for images) — `hold_seconds` still applies as a hard ceiling.
  - When the message has no media, the coordinator falls back to `self.effects[self.idx]` (the current rotation behavior). No behavioral change for SMS-only messages.

**Wire format (admin UI + browser)**

- The `/messages` admin view shows a small thumbnail strip below each message with media (admin fetches the proxy URL with the existing API key auth). Click a thumbnail for full-size in a modal.
- The browser preview's `MessageView` already round-trips `to_dict()` → `js.JSON.parse`; the media list is read by the preview's `MediaCycler` (a browser-suitable stand-in for the Pi-side `MediaCycler`, sharing the same constructor signature). The browser preview becomes a useful place to QA media rendering without touching the Pi.

## Capabilities

### New Capabilities

- `mms-media-support`: Twilio MMS webhook ingestion captures `NumMedia` + `MediaContentType{i}` + `MediaUrl{i}` and responds to Twilio with 200/TwiML immediately. A background thread then downloads each attachment via Twilio Basic Auth, copies the bytes to S3 under `media/images/{YYYY-MM}/` or `media/videos/{YYYY-MM}/` (per content type) before the Twilio retention window expires, persists the resulting `Message` (text + media list) to S3 + SQLite, and publishes the `MessageEnvelope` over MQTT once all uploads complete. The `Message` wire grows an optional `media: list[{type, url}]` array (defaulting to `[]` for backward compat). The Pi and browser fetch media through a new authenticated Flask endpoint `GET /api/media/<path:s3_key>` that returns a 302 redirect to a freshly-signed S3 URL. The `EffectsCoordinator` routes messages with media to a per-message `MediaCycler` (constructs `ImageDisplay` or `VideoDisplay` per item based on mime, cycles every `max(10 s, media_duration)` up to `hold_seconds`, then yields to the rotation; bad-codec items are dropped from the cycler's list with a WARNING — empty list falls back to the rotation). MMS-only path — pure SMS messages get no new surface.
- `image-display-pattern`: Rename `lib_shared/patterns/png_display.py` → `image_display.py` and the class `PngDisplay` → `ImageDisplay`. Same palette-based rendering through the indexed `Bitmap`/`Palette` pipeline; the load path drops the alpha-channel trick (JPEG has no alpha; GIF's anim is collapsed to single frame, same as PNG). Supports PNG/JPEG/GIF/WebP via PIL — same module deps it already has. **`ImageDisplay` and `VideoDisplay` are removed from the canonical effects registry** (they were listed-but-disabled entries in the post-PR-53 JSON, pointing at operator-supplied content paths that are now superseded by MMS). `ImageDisplay` lives on as the inner renderer consumed by `MediaCycler` (see `mms-media-support`); the rotation carries only the 5 non-media effects. `VideoDisplay` is untouched in this capability.

### Modified Capabilities

_None. There are no prior capability requirements on this repo (the `openspec/specs/` tree is empty); every behavior change above is a new surface._
