## Why

The sign's runtime behavior — which effects are in the rotation, how long a message holds, how fast the scroller moves, the boot-splash duration, the intro/outro fade speed, how many recent messages the idle rotation can pull from — is hard-coded in `EffectsCoordinator` constructor defaults and the hard-coded effect list in `heart-matrix-controller/main.py`. Operators have to SSH to the Pi, edit a Python file, and reboot the device to change any of it. The Flask admin's `/settings` page (Tailwind playful redesign) shows sign name, timezone, rendering mode/speed/color, allowed senders, and filter rules — but those are stored, fetched, and re-rendered as form fields; the device ignores them (it uses its own constructor defaults), and the field for `rendering.color` and `rendering.speed` are not wired to the actual scroller or any effect.

The wire path for runtime config already exists end-to-end (`type="config"` `MessageEnvelope` → SQLite → S3 snapshot → MQTT publish → `MessageManager._handle_config` → in-memory `SignConfig.update_from_dict`), and the browser-side `MessageManager` reads the same config on seed. What's missing is the **content** of that config — the fields that meaningfully change what the sign does at runtime — and a UI to edit them without a redeploy.

This change adds two families of runtime settings to `SignConfig`, plumbs them through Flask ↔ MQTT ↔ the device's `EffectsCoordinator` and `Scroller`, and surfaces them in the admin UI:

1. **`effect_settings`** — a single block that groups everything the `EffectsCoordinator` needs: the **full canonical list of effects with per-entry enabled flags** (`effects: list[{"name": str, "enabled": bool}]`), and the small set of timing + rotation parameters that map to the coordinator's constructor (fade, hold, intro, idle, recent_count). The list is the FULL set of known effect classes (Hyperspace, VideoDisplay, PngDisplay, Honeycomb, Flame, Fireworks, NightSky — 7 entries), each with an `enabled: bool` flag the operator toggles. The rotation is built by iterating the list in order and including only `enabled: true` entries; the order in the list IS the rotation order. The technical parameters operators don't need to reason about (`fade_step`, `gamma`) stay as code defaults. The grouping keeps non-effects concerns (filters, senders, sign name, timezone, text rendering) out of the coordinator's constructor. Future additions to the coordinator (transition modes, etc.) land in the same block — its name is "Effects" because it's everything the effects subsystem consumes, not just "pacing" or "rotation."
2. **`text_settings`** — a single block for the on-screen text rendering parameters: `frame_delay` (scroll speed), `offset_seconds` (two-line offset), `color`, and a `text_effect` field (`"scroll"` today, with room to add `"swirl"`, `"bounce"`, etc. later without a schema change). The block is named `text_settings` (not `scroller_settings`) because the scroller is just one text effect — future text effects (swirl, bounce) will share the same block and may use different fields. The issue explicitly asks for the structure even though only `scroll` is implemented in v1. This block **replaces** the old `SignConfig.rendering` block (the v1 `RenderingSettings` with `speed`, `color`, `mode`): the v1 block is being removed entirely, with the text color moving into the new `text_settings` block and the rest of `rendering` (speed, mode) being dropped (the operators never actually used them — the device used its own scroller constructor defaults).

**Why a single `effect_settings` block and not separate `effects` + `pacing` top-level fields?** The coordinator consumes both — pairing them in one object means the coordinator's constructor takes one focused argument, not "the whole `SignConfig`." Future additions to the coordinator (e.g. a `transition` mode, an `intro_splash` toggle) land in the same block without growing `SignConfig`'s top-level surface. The coordinator's narrower knowledge (no filters, no sign name, no timezone) is a real win — the same shape that lets a future `ScrollerSettings`-like block be consumed by a different class without bleeding the rest of `SignConfig` in.

While we're touching this code, the issue flags two cleanups that fall naturally out of the new wiring and one config field that the Pi no longer needs:

- **Remove `tz_offset_mins`** from `SignConfig` and the message-display path. The Pi can use `zoneinfo.ZoneInfo` for the offset; the offset was a CircuitPython workaround. Compute the offset on the server from the IANA `timezone` at read-time, not at config-define-time.
- **Remove `EffectsCoordinator.request_message`** and the `recent_provider` callable. Both the browser and the Pi already have a `MessageManager`; the coordinator can read its internal recent-bodies deque from the manager directly, so the lambda-and-`on_message` indirection in `heart-matrix-controller/main.py` (and the parallel internal-deque path in the browser) collapses to one path. If the browser and Pi end up using the same mechanism, the callable parameter becomes a single `message_manager` argument.
- **Investigate Wi-Fi credential management** — out of scope for v1 implementation, but a note in the design captures the constraint (the Pi OS owns Wi-Fi; a bad write locks the device out unless we test before commit) and parks it as a follow-up change.

The config shape is changing in a way that breaks any stored `version: 1` config (the `tz_offset_mins` field disappears; the new `effect_settings` and `scroller` blocks appear). This change adds a `version` flag on the config and a small `MIGRATIONS` registry that brings older payloads up to the current version on read, so existing stored configs and suppression rules in S3 / SQLite are preserved across the upgrade without manual operator action. See the **Migration & Versioning** section below.

## What Changes

- `SignConfig` (in `lib_shared/models.py`) gains two new top-level fields: `effect_settings: EffectsSettings` (a single block containing the full effects list with per-entry `enabled` flags, the four timing durations, and `recent_count`) and `text_settings: TextSettings` (frame_delay, offset_seconds, color, text_effect). The old `effects` and `pacing` top-level fields are gone — they're now nested under `effect_settings`. The old `rendering` top-level field is **removed entirely** — the new `text_settings` block replaces it. All new fields have `from_dict` / `to_dict` round-trip support and the same `threading.RLock` guard as the existing fields. `tz_offset_mins` is removed from the model and all readers; the offset is computed at read-time via `zoneinfo` from `timezone`.
- `EffectsSettings` and `TextSettings` (both in `lib_shared/models.py`, alongside `SignConfig` — not separate files) are two small classes. `EffectsSettings` groups the coordinator's inputs: `effects: list[{"name": str, "enabled": bool}]` (the full canonical set with per-entry toggle), `fade_seconds: float`, `hold_seconds: float`, `intro_seconds: float`, `idle_seconds: float`, and `recent_count: int`. `TextSettings` groups the text rendering inputs: `frame_delay: float`, `offset_seconds: float`, `color: int`, `text_effect: str` (in v1, only `"scroll"` is allowed; the validator raises on any other value). The `recent_count` field is surfaced in the admin UI per operator feedback — operators want to control "how many recent messages the idle rotation can pull from" without redeploying. The other two `EffectsCoordinator` parameters that aren't surfaced (`fade_step`, `gamma`) stay as code defaults. The `_DEFAULT_EFFECTS_LIST_FULL` constant (a list of `{"name", "enabled"}` dicts) lives in `models.py` (shared between Flask and the Pi), and `TEXT_EFFECTS` lives on `TextSettings` for the same reason — keeping everything in one file matches the small-surface-area of this module and avoids a 20-line file for a 20-line class. The default `_DEFAULT_EFFECTS_LIST_FULL` covers all 7 known effects (Hyperspace, VideoDisplay, PngDisplay, Honeycomb, Flame, Fireworks, NightSky) with the 5 historically-defaulted effects set to `enabled: true` and the 2 asset-dependent effects (VideoDisplay, PngDisplay) set to `enabled: false` — matches the historical 5-effect rotation visual out of the box while keeping the full set visible for the operator to enable later. The old `RenderingSettings` class (which backed the v1 `rendering` block with `speed`, `color`, `mode`) is **removed** — its only use site was `SignConfig.rendering`, which is also being removed.
- `EffectsCoordinator.__init__` accepts an `effect_settings: EffectsSettings` argument (NOT the full `SignConfig` — the coordinator shouldn't know about filters, senders, sign name, or timezone). The existing per-field keyword arguments remain as per-field overrides; the default is to read from `effect_settings`. The recent-message source becomes a single `message_manager` argument (a `MessageManager` or any object with a `get_messages(limit, suppress=True)` method) — both the Pi and the browser pass their existing `MessageManager`, and the lambda + on_message wiring in `heart-matrix-controller/main.py` (and the parallel internal-deque path in the browser) collapses to one path. `request_message(text)` is removed; the coordinator's `_recent` deque is fed exclusively by the manager.
- `heart-matrix-controller/main.py` builds the effect list at startup by **iterating the configured `effect_settings.effects` list in order and including only entries with `enabled: true`**, instantiating the matching class for each (skipping any that fail to construct, with a log warning). This is the only mechanism that controls the rotation — there's no hard-coded literal list. The device's `_EFFECT_CLASSES` map (effect class names → classes) is the canonical set; entries in `effect_settings.effects` whose `name` is not in `_EFFECT_CLASSES` are logged and skipped. The default `effects` list is the hard-coded constant `_DEFAULT_EFFECTS_LIST_FULL` in `lib_shared/models.py` (7 entries, 5 enabled + 2 disabled). `settings.toml` is not the source of truth for the rotation (per operator feedback — the existing device's `settings.toml` is small and the defaults should stay in code where they can be reviewed alongside the effect classes).
- The settings page (`heart-message-manager/templates/settings.html`) gets two new sections. The first is one big **Effects** section with two sub-sections: **Effects List** (a checkbox per known effect in the canonical order from the device's `_EFFECT_CLASSES` map, excluding `Heartbeat` which is the boot-splash; the list is read directly from `cfg.effect_settings.effects` — no separate `/api/effects` fetch) and **Settings** (five labeled sliders: fade, hold, intro, idle, recent — each with a human-readable label and a number input next to the slider; `recent_count` is surfaced per operator feedback). The sub-section is called "Settings" (not "Pacing" and not "Behavior") because future fields added to the block (transition modes, etc.) won't all be pacing/behavior, and "Settings" inside "Effects" is clear about what it's configuring. The second section is **Text** (scroll-speed slider + a text-color input + a static "scroll" text-effect dropdown for v1, with the dropdown disabled to signal that more options are coming). The Text section replaces the old "Scrolling" section name and absorbs the text-color field that used to live under "Rendering Defaults" (which is being removed entirely — the old `rendering` block is gone). The form is still a single POST that PUTs to `/api/config`.
- The Flask `PUT /api/config` endpoint validates the new fields (effects must be a list of `{"name", "enabled"}` objects where `name` is from the device's known effect set, `enabled` is a boolean; behavior fields must be non-negative numbers, recent_count must be a positive integer, text fields must be non-negative, text_effect must be one of an enum). Unknown `name` values return 400 with a per-field error message. The wire shape is unchanged otherwise except for the new blocks and the removal of `rendering` / `tz_offset_mins`.
- ~~A new `GET /api/effects` endpoint returns the canonical list of effect class names available on the device~~ — **dropped**. The admin UI reads the effects list directly from `cfg.effect_settings.effects` (already returned by `GET /api/config`); a separate endpoint would duplicate state. The canonical set of "what effect classes exist" lives in `lib_shared/models.py` as `_DEFAULT_EFFECTS_LIST_FULL` (the default) and on the device as `_EFFECT_CLASSES` (the class map); the UI treats the `effects` list as authoritative for which entries to render.
- `lib_shared/messages.py` no longer reads `tz_offset_mins` from the config; it computes the offset from the IANA `timezone` via `zoneinfo.ZoneInfo` at the call site. `heart-message-manager/sqlite.py` no longer needs to recompute the offset on every config write.
- A new `lib_shared/config_migrations.py` module houses a `MIGRATIONS` registry: a dict `{1: _v1_to_v2, ...}` mapping source version to a `(dict) -> dict` callable that takes a payload at version N and returns a payload at version N+1. `migrate(d, current_version)` chains the calls. `SignConfig.from_dict` and `SignConfig.update_from_dict` call `migrate` at the top as defense-in-depth, so any stored v1 config (in SQLite, in S3, or arriving over MQTT) is transparently upgraded to v2 on read. The Flask PUT handler also runs migrations on the incoming payload so the wire shape is always normalized to v2 before persistence. **Critically, the server runs the migration on startup** (alongside the existing "rebuild-from-S3 on startup" path in `heart-message-manager/sqlite.py`): if the S3-stored config's version is older than `CURRENT_VERSION`, the server runs `migrate()`, writes a new S3 entry, updates SQLite, and pushes the migrated config to MQTT. This means the running code only ever sees the current version — it doesn't have to be backward-compatible with old shapes for months waiting for an operator to click "Save."
- Wi-Fi credential management is **out of scope for v1 implementation**; the design documents the constraint and parks a follow-up change that adds a `nmcli connection modify` + `nmcli connection up` flow with a pre-commit `nmcli device wifi connect` test.

### Migration & Versioning

This is a breaking change to the config shape. The `version` field on `SignConfig` (already present, currently defaulting to 1) is bumped to 2, and a `MIGRATIONS` registry handles the upgrade path. The pattern is small and well-established (it's the same shape Rails' `data_migrations`, Django's `migrations/`, and Pydantic's `model_validator(mode="before")` use, but minimal — for this project a full ORM migration framework would be overkill).

**What gets preserved on upgrade from v1 to v2:**

- `filters` (suppression rules) — unchanged between v1 and v2.
- `senders` — unchanged.
- `sign`, `rendering`, `timezone` — unchanged.
- The on-disk message list (`messages.json` in S3) is not part of the config; the migration doesn't touch it.
- The Pi's local in-memory `MessageManager` ring buffer is rebuilt from the seed fetch; the migration is server-side.

**What changes on upgrade from v1 to v2:**

- `tz_offset_mins` is dropped from the stored config (it was a CircuitPython-era field; the offset is now computed at read-time).
- `rendering` is dropped from the stored config (the old `RenderingSettings` block — `speed`, `color`, `mode` — is being removed entirely; the new `text_settings` block replaces it with `frame_delay`, `offset_seconds`, `color`, `text_effect`).
- `effect_settings` is added with the hard-coded defaults (`effects` = the full 7-entry list, with 5 historically-defaulted effects set to `enabled: true` and `VideoDisplay` / `PngDisplay` set to `enabled: false`; the four timing durations and `recent_count` match the historical `EffectsCoordinator` defaults).
- `text_settings` is added with the hard-coded defaults (matching the historical `Scroller` constructor defaults for `frame_delay` / `offset_seconds` / `color`, and `"scroll"` for `text_effect`).
- `version` is bumped from 1 to 2.

**When the upgrade happens:**

- On server startup, after the existing "rebuild-from-S3 on startup" step in `heart-message-manager/sqlite.py` reads the latest config from S3, the server runs `migrate(...)` on it. If a migration ran (i.e. the stored version is older than `CURRENT_VERSION`), the server immediately:
  1. Writes a new S3 entry at the current version, replacing the old one.
  2. Updates the local SQLite cache to the migrated config.
  3. Publishes a `type="config"` envelope to MQTT so any connected devices (Pi, browser preview) pick up the migrated config on their next MQTT read.

  This is the central design decision: the running code never has to be backward-compatible with old shapes for months waiting for an operator to click "Save" on the settings page. The startup migration is idempotent — if the S3 config is already at `CURRENT_VERSION`, nothing is written and nothing is published.

- The Flask `GET /api/config` endpoint returns the migrated config (Flask normalizes on read; the migration would happen inside `from_dict` if the row is then reloaded, but the GET path runs it explicitly so the response is always at the current version).
- The Flask `PUT /api/config` endpoint runs the migration on the incoming payload before constructing the `SignConfig`. The saved SQLite row is always at `CURRENT_VERSION`.
- The device's `MessageManager._handle_config` calls `update_from_dict`, which runs the migration at the top. A v1 payload arriving over MQTT (e.g. from a still-running v1 device, or a stale MQTT message from before the server's startup migration) is upgraded to v2 in the device's in-memory config.

**Why a registry and not a versioned file format or a Pydantic-style `Union[ConfigV1, ConfigV2]`?** The shape is small (one struct, ~8 fields), the migration list is short (one entry: v1 → v2), and the project doesn't have a database schema — the config is a single JSON blob. A registry that takes the current version and chains upgrade functions is the simplest mechanism that scales to v3+ without changing the read path. The alternative (parse by version, dispatch to a `ConfigV1` or `ConfigV2` class) doubles the type surface and doesn't help when v3 lands.

**Why a `version` flag and not just a "drop unknown fields" policy?** Dropping unknown fields silently loses data — a v1 client receiving a v2 payload would lose `effect_settings`. The `version` flag makes the wire shape explicit: each consumer (Flask, device, browser preview) knows the shape it expects, and the migration registry brings older shapes forward. This is the same pattern that lets a v1 device and a v2 device coexist in the field during a rolling upgrade — the v2 device's `migrate(v1_payload)` returns a v2 payload; the v1 device simply never sees a v2 payload (because the Flask side normalized it on write).

## Capabilities

### New Capabilities

- `sign-runtime-config`: `SignConfig` carries an `effect_settings` block (full canonical effect list with per-entry `enabled` flags, plus fade/hold/intro/idle durations and `recent_count`) and a `text_settings` block (frame_delay, offset_seconds, color, text_effect). The old `rendering` block is REMOVED. The full config persists to SQLite, snapshots to S3, and publishes to MQTT as a `type="config"` `MessageEnvelope`. The Pi reads the config on boot and on every config message, rebuilds the effect list from `effect_settings.effects` (filtering by `enabled: true`), and constructs `EffectsCoordinator` and `Scroller` from the new blocks. The default `effect_settings.effects` is a hard-coded constant `_DEFAULT_EFFECTS_LIST_FULL` in `lib_shared/models.py` (all 7 known effects, 5 enabled + 2 disabled); the default `text_settings` values match the historical `Scroller` constructor defaults. A `MIGRATIONS` registry in `lib_shared/config_migrations.py` brings older `version: 1` payloads forward to v2 transparently on read, AND the server runs the migration on startup so the running code only ever sees the current version.
- `sign-settings-ui`: the admin UI's `/settings` page surfaces the new fields in one big **Effects** section (with **Effects List** and **Settings** sub-sections — "Settings" is the new generic name, leaving room for future non-timing fields) and a **Text** section (formerly called "Scrolling"). The old "Rendering Defaults" section is REMOVED — its text-color field has moved into the new Text section. The form is still a single POST that PUTs to `/api/config`; the new fields are validated server-side and round-trip through SQLite/S3/MQTT like the existing ones.
- `effects-coordinator-cleanup`: `EffectsCoordinator.request_message` is removed; the coordinator's recent-message source is the `MessageManager` (a single path used by both the Pi and the browser), so the `recent_provider` callable becomes a single `message_manager` argument. `EffectsCoordinator` takes an `EffectsSettings` argument (not the full `SignConfig`) so the coordinator doesn't gain knowledge of filters / senders / sign / timezone. `SignConfig.tz_offset_mins` is removed; the timezone offset is computed at read-time from the IANA `timezone` via `zoneinfo`. The `SignConfig.rendering` field (and the `RenderingSettings` class) is REMOVED — the new `text_settings` block replaces it. The wire path is unchanged (still `type="config"` envelopes with the same `from_dict`/`to_dict` shape, plus the new blocks, minus `tz_offset_mins` and minus `rendering`).

### Modified Capabilities

- `sign-preview-rendering` (in `openspec/changes/add-sign-preview-rendering`): the browser preview's `MessageManager` already seeds from `/api/config` and applies config updates via `MessageManager._handle_config`. Once the new fields exist, the browser preview's `EffectsCoordinator` and `Scroller` will also read from the new config — the same way the Pi does. No new preview-only behavior; the preview just gets the new fields for free because the seed path is shared. The preview's `request_message` is removed by this change; the preview's existing `setInterval(pollLatestMessage, 3000)` already pushes new bodies into the coordinator via `request_message` — that call site switches to the new mechanism (`message_manager.dispatch` or a new `message_manager.feed_to_coordinator(coordinator)` method). The new mechanism is identical between browser and Pi.
