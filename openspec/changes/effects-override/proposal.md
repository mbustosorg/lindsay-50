## Why

Adding a new background pattern to the Pi currently requires coordinated edits across `lib_shared/effects_factory.py` (hardcoded name→class branches), `lib_shared/models.py` (`_DEFAULT_EFFECTS_LIST_FULL`), and `heart-message-manager/main.py` (`_KNOWN_EFFECT_NAMES`). A recently-merged recipe ([docs/adding-patterns.md](../../../docs/adding-patterns.md)) added a `local_effects` kwarg on `EffectsCoordinator` as a transitional mechanism — but the user has rejected that transitional path entirely. We can generalize this to a config-file-driven registration system: adding a pattern becomes a JSON edit, and Pi-local development becomes a gitignored override file in a dedicated folder, with REPLACE semantics.

The JSON schema mirrors the `EffectsSettings` dataclass shape — every field on the class is a field on the JSON, with `module` + `class_name` added to each effect dict so the factory can resolve them dynamically. One file, one env var, one override path, one wire-strip rule. No invented metadata (description, category, default_enabled, requires_assets, asset_paths) — the class IS the schema. As part of this change we also **revert** the recipe's `local_effects` kwarg additions, fold `effects_factory.py` into `effects_loader.py` (one cohesive module), and delete `_DEFAULT_EFFECTS_LIST_FULL` outright — clean break, no transitional shims.

## What Changes

- **NEW** `lib_shared/config/effects_settings.json` (in git, canonical) — declares every known effect with `name`, `module`, `class_name`, `enabled` plus the dataclass fields: `fade_seconds`, `hold_seconds`, `intro_seconds`, `idle_seconds`, `recent_count`. Lives in the new `lib_shared/config/` folder alongside future canonical config files (`config.json`, `seed_messages.json`) so all in-git config files are colocated and easy to find. All 7 effects are included: Fireworks, Flame, NightSky, PngDisplay, VideoDisplay, Honeycomb, Hyperspace. The asset-dependent ones (`PngDisplay`, `VideoDisplay`) carry `enabled: false` so the Flask admin UI surfaces the full toggle surface but the rotation is safe on a fresh checkout.
- **NEW** `lib_shared/effects_loader.py` — override-aware loader. `load_effects_settings()` returns the override if present, else the canonical. `is_effects_settings_override_active()` returns True if the override file exists. `reset_effects_settings()` clears the cache for tests. Plus the factory `make_effect_class(name)` (folded in from the deleted `effects_factory.py`). REPLACE-only semantics.
- **NEW** `config_overrides/` folder at the repo root (gitignored as a single entry). The override file lives at `config_overrides/effects_settings.json` and uses the same schema as the canonical. Future overrides (`config.json`, `seed_messages.json`) will live in the same folder, all covered by one gitignore line.
- **DELETE** `lib_shared/effects_factory.py`. The factory is folded into `lib_shared/effects_loader.py` — `make_effect_class()` becomes a function in the loader module. There is no separate factory module after this change. The factory uses a dynamic loader loop (`importlib.import_module` + `getattr`) over the loaded config. On-demand import (no numpy/cv2/PIL at module load) is preserved.
- **REVERT** `lib_shared/effects_coordinator.py` — undo the `local_effects` kwarg additions from the recipe (added to `__init__`, `bind()`, and `_tick_inner`'s re-append branch). The override file is the new mechanism; the transitional kwarg never sees production use. The recipe was merged with these lines in place; this change reverts them as part of the broader JSON-driven refactor.
- **MOD** `lib_shared/message_manager.py` — when `is_effects_settings_override_active()` is True, pop `payload["effects_settings"]` if present before calling `SignConfig.update_from_dict`. Top-level `text_settings` and `filters` still come from the wire as normal.
- **MOD** `lib_shared/models.py` — **DELETE** `_DEFAULT_EFFECTS_LIST_FULL` outright. No backward-compat alias, no derived property, no module-`__getattr__` shim. Consumers MUST call loader functions (`load_effects_settings()` / `effects_loader.reset_effects_settings()` for tests). The 3 test files that monkey-patched the alias get rewritten to inject loader fixtures. The `EffectsSettings` dataclass keeps its current shape but its constructor reads canonical defaults from `effects_settings.json` on first call (the dataclass field defaults stay as fallbacks).
- **REVERT** `heart-matrix-controller/main.py` — undo the `local_effects=[...]` kwarg at the `EffectsCoordinator(...)` construction site (line 136). The recipe added 1 line here; this change reverts it.
- **MOD** `heart-message-manager/main.py` — derive `_KNOWN_EFFECT_NAMES` from the loaded config at startup. The hardcoded frozenset disappears. Pass `effects_settings = load_effects_settings()` to the admin UI routes (both `/settings` and `/playful/settings`).
- **MOD** `heart-message-manager/templates/settings.html` and `templates/playful/settings-playful.html` — iterate `effects_settings.effects` for the per-row metadata and the wire state (`enabled`). The top-of-form `fade_seconds` / `hold_seconds` / `intro_seconds` / `idle_seconds` / `recent_count` inputs read from `effects_settings` (falling back to `cfg.effects_settings.*` once a wire envelope has arrived).
- **MOD** `.gitignore` — add `config_overrides/` (single entry covers all current and future override files).
- **MOD** `tests/effects_loader_test.py` (folded factory tests into the loader test file per the "no separate factory module" decision), `tests/effects_coordinator_test.py`, `tests/effects_settings_test.py` — use the new loader; drop `local_effects` tests.
- **MOD** `tests/test_boot_config_endpoint.py`, `tests/test_auth.py`, `tests/test_sign_status_endpoint.py` — rewrite to inject loader fixtures via `effects_loader.reset_effects_settings()` after setting `_FAKE_CONFIG`. No monkey-patching of `_DEFAULT_EFFECTS_LIST_FULL` (the alias no longer exists).
- **MOD** `docs/adding-patterns.md` — rewrite for the new mechanism. The recipe becomes "edit `config_overrides/effects_settings.json` and you're done" — much shorter than the current 531-line doc.

**Wire shape is unchanged.** Today: `effects_settings = {"effects": [{"name": "X", "enabled": bool}, ...], "fade_seconds": ..., "hold_seconds": ..., "intro_seconds": ..., "idle_seconds": ..., "recent_count": ...}`. After: identical (with `module` and `class_name` added per-effect, which are stripped before constructing `EffectsSettings`). The Flask admin derives the wire shape from the loaded JSON. The Pi parses it the same way. The only difference: with the override active on the Pi, the wire's `effects_settings` block is dropped on receipt.

## Capabilities

### New Capabilities

- `effects-override`: The mechanism for declaring the canonical effects settings in `lib_shared/config/effects_settings.json`, the override file at `config_overrides/effects_settings.json`, the override-aware loader, and the rule that the Pi strips the wire's `effects_settings` block when the override is active. Covers factory registration (Flask + Pi see the same dynamic name→class map), Flask admin UI rendering (templates iterate the loaded effects), and Pi-local development for the entire `EffectsSettings` block.

### Modified Capabilities

(none — no existing spec covers the effects registry or rotation)

## Impact

- **Affected code**: `lib_shared/effects_factory.py` (DELETE), `lib_shared/effects_loader.py` (NEW — absorbs factory), `lib_shared/effects_coordinator.py` (REVERT recipe additions), `lib_shared/message_manager.py` (wire-strip on override), `lib_shared/models.py` (REMOVE `_DEFAULT_EFFECTS_LIST_FULL`, `EffectsSettings.__init__` reads from loader), `heart-matrix-controller/main.py` (REVERT recipe addition), `heart-message-manager/main.py`, `heart-message-manager/templates/settings.html`, `heart-message-manager/templates/playful/settings-playful.html`, `.gitignore`, `docs/adding-patterns.md` (rewrite for new mechanism — recipe's transitional path is gone), ~6 test files.
- **Wire format**: unchanged (with `module` + `class_name` stripped on consumption).
- **Public APIs**: `lib_shared.effects_loader.make_effect_class(name)` is the new home of `make_effect_class` (was `lib_shared.effects_factory.make_effect_class`); signature unchanged. NEW: `load_effects_settings()`, `is_effects_settings_override_active()`, `reset_effects_settings()` — all live in `lib_shared.effects_loader`. `_DEFAULT_EFFECTS_LIST_FULL` is REMOVED — no replacement, no alias.
- **Operator workflow**: existing operators see no behavior change. Operators who create `config_overrides/effects_settings.json` opt into local-dev mode (the Pi stops accepting the wire's `effects_settings` block and uses its own JSON instead — including the effects list AND the timing fields).
- **Risk**: low. The loader refactor is a behavior-preserving code cleanup (same 7 effects resolve the same way, just from one module instead of two). The override-active wire-strip is a new opt-in path. Three clean breaks: recipe revert, factory→loader merge, and `_DEFAULT_EFFECTS_LIST_FULL` deletion — none have other callers outside the files being edited in this PR.
- **Out of scope** (separate changes): `config.json` override (text_settings, filters, senders), `seed_messages.json` override (offline message seeding), full "local dev mode" where all external inputs are stubbed, plugin/decorator discovery system, MERGE semantics for the override.
