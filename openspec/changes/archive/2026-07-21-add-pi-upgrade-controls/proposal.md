## Why

Managing Pi upgrades today is binary: either auto-upgrade runs on Flask deploys (the loader + one-shot `check-for-update` MQTT hint + `/api/sign/boot-config` HTTP fetch) or the operator SSHes in by hand. There is no way from the Flask admin UI to pin the Pi to a specific version or to issue a force-upgrade / restart / shutdown without physical access. The operator is not always on-site, and SSHing to recover from a failed upgrade is the exact failure mode the original `self-upgrading-matrix-controller` change was trying to eliminate. We need operator-facing controls that ride on top of the existing loader + SHA-exchange mechanism.

Two concrete gaps this change closes:

1. **No version pinning.** If the Pi is healthy at version `abc` but Flask just deployed `def` and the auto-upgrade handler triggers, the operator has no way to keep the Pi at `abc` short of disabling auto-upgrade in code and redeploying.
2. **No recovery commands.** Force-upgrade, restart, and shutdown all require SSH today. A force-upgrade is the natural recovery path when the Pi auto-updated to a broken version; restart/shutdown are needed after settings changes that require a reboot (e.g. Wi-Fi reconfiguration).

This change simplifies the existing flow at the same time:

- A new endpoint `GET /api/sign/settings` returns just the short version + timezone. The existing `/api/sign/boot-config` endpoint is **kept unchanged** so the existing Pi can still upgrade through it (legacy path, untouched). New code uses the new endpoint; the legacy endpoint remains for backward compatibility.
- **The long form is dropped everywhere.** The loader's `git worktree add` accepts a short SHA as long as it's unambiguous in the repo (per `git-worktree(1)` and the comment at `loader.py:174`). The worktree directory naming convention (`v-<short_sha>`), the `atomic_swap` symlink target, and the local-vs-expected comparison all work with short SHAs — the equality check `local == expected` becomes `short_sha(local) == expected`. No long form needed on the wire or in `SignSettings`.
- **The `check-for-update` MQTT hint is KEPT during the transitional period** (this change). The new `/api/sign/settings` endpoint returns the resolved target on every boot, which makes the hint technically redundant — but we keep it registered alongside the three new handlers so currently-deployed Pis (which only understand `check-for-update`) can roll forward through this change. The `check_for_update.py` handler module is **not** deleted; Flask's startup **continues** to publish the one-shot `check-for-update` envelope. A follow-up change will delete the legacy path once all deployed Pis are confirmed to be on the new code.
- **`sign.target_version` is always concrete on the wire.** Flask resolves operator-pin vs Flask-self before persisting and before publishing — the Pi never sees `null` and never has to compute the "track Flask" fallback itself. The UI input is nullable (operator's intent), but the persisted value is always a short version string.

The status data needed to drive the UI (Pi's running SHA, Flask's running SHA, last tick, MQTT-connected state) is already on the wire thanks to the now-merged `add-sign-status-reports` change (both Pi and Flask publish `StatusSnapshot` to `MQTT_STATUS_TOPIC` on a 5s cadence; Flask exposes the latest via `GET /api/sign-status`; browser hydrates on load and subscribes via a second `createMqttWsClient`). This change consumes those fields directly from the existing pipeline.

## What Changes

- **Add `GET /api/sign/settings`** returning `{"target_version": "<7-char>", "timezone": "US/Pacific"}`. Two fields, both always concrete on the wire. Auth: same `X-API-Key` as `/api/config`. The Pi's loader calls this on boot with a 5-second timeout; any failure falls through to running the current version.
- **Keep `/api/sign/boot-config` unchanged.** Existing Pis continue to call it. New Pis call `/api/sign/settings`. No deletion, no migration of existing consumers.
- **Keep `check-for-update` MQTT command for the transitional period.** Today Flask publishes a one-shot `action=check-for-update` envelope at startup so the running Pi compares its `LINDSAY50_ACTIVE_SHA` against Flask's expected SHA. With `/api/sign/settings` returning the resolved target on every boot, the new code path makes the hint technically redundant for new Pis — but we keep the existing `check_for_update.py` handler registered alongside the three new handlers so currently-deployed Pis (which only understand `check-for-update`) can roll forward through this change. Flask's startup **continues** to publish the one-shot `check-for-update` envelope. A follow-up change drops the legacy path once all deployed Pis are confirmed to be on the new code.
- **Flask publishes `type=config` on boot.** To keep the Pi's in-memory state in sync with the operator's last save (not just the loader's boot-time check), Flask publishes the latest `type=config` envelope once on startup. If the persisted `SignConfig` hash matches the last-published hash (kept in a sidecar file), the publish is skipped — keeps MQTT traffic flat on routine deploys that don't touch settings.
- **`SignSettings` adds one field:** `target_version: str`. `target_version` defaults to the Flask running SHA (short form) at construction time when not specified (server-side resolution; the field is never empty on the wire). `timezone` is NOT added to `SignSettings` — it stays at the top level of `SignConfig` where it already lives.
- **No schema bump, no migration.** `SignConfig.CURRENT_VERSION` stays at **2**. The new `sign.target_version` field is forward-compatible (pre-change Flask ignores the unknown key) and backward-compatible (pre-change code reads the field via the dataclass default if missing).
- **New Settings UI section — "Pi Upgrade Control".** Three read-only version fields plus one editable input. Lives as a sub-section of the existing `/settings` page (no new page):
  - **Flask version** — read-only display of the Flask server's currently-running commit SHA (short form). Sourced from the latest status snapshot consumed via the `add-sign-status-reports` browser→MQTT-WS flow (5s cadence). When no snapshot has been received yet, the field shows "No status received yet".
  - **Pi version** — read-only display of the Pi's running SHA (short form). Same source as Flask version. When no snapshot has been received yet, the field shows "No status received yet".
  - **Target Pi version** — editable input. The operator types a 7-character short SHA to pin the Pi; empty input means "track Flask version" (which Flask resolves to its own running SHA before publishing). A "Clear" button alongside the input returns the field to the empty/track-Flask state. The hint text below the input reads "Currently pinned to: `<short-sha>`" when set, or "Tracking Flask version — currently `<short-sha>`" when empty. Persisted as `sign.target_version` on `SignConfig`. Saves through the existing `POST /api/config` endpoint — no new endpoint, no new model.
- **Pi-side `AUTO_UPDATE` continues to be a `settings.toml` key** (not a config field, not a UI control). The loader already reads `AUTO_UPDATE` from `heart-matrix-controller/settings.toml` (see `loader.py:833`) — this change does not modify that key. The operator flips it by SSHing in and editing the file (or via the systemd unit `Environment=AUTO_UPDATE=true` env override). A **future change** will add a Pi-side `settings.json` override file (mirroring `effects_settings.json`) so the operator can flip the flag from the Pi without re-deploying settings.toml.
- **New commands** issued via the existing `type=command` MQTT envelope + `MessageManager.dispatch` handler mechanism (introduced in `self-upgrading-matrix-controller`):
  - **`force-upgrade`** — tells the Pi to attempt an upgrade to the current effective target NOW, regardless of `AUTO_UPDATE`. Issued from a button on the new Settings section. The Pi handler `os.execvpe`s into the loader, which performs the standard SHA-check + stage + probe + swap (or no-op if already at target).
  - **`restart`** — reboots the Pi. Issued from a button. The handler calls `subprocess.run(["sudo", "reboot"], check=False)` (loader already runs as root via systemd). After reboot, the loader runs on startup and re-reads the resolved target via `GET /api/sign/settings`.
  - **`shutdown`** — powers off the Pi. Issued from a button. The handler calls `subprocess.run(["sudo", "shutdown", "-h", "now"], check=False)`. After power-on, the loader runs and decides whether to upgrade.
- **Flask-side status reporting** (already covered by the merged `add-sign-status-reports` change). Flask publishes its runtime SHA on `MQTT_STATUS_TOPIC` on the same 5s cadence with a `source: "flask"` discriminator.

### Out of Scope (parked for follow-up changes)

- A separate MQTT topic for discrete events (per the existing parking lot in `add-sign-status-reports`).
- A **Pi-side `settings.json` override file** for `AUTO_UPDATE` (mirrors the existing `effects_settings.json` mechanism). v1 uses the existing `AUTO_UPDATE` key in `settings.toml`. The override file would let the operator recover from a bad deploy without SSH + redeploy.
- Pi-side persistent version pinning beyond `sign.target_version`.
- Per-SHA rollback UI (the existing `heroku rollback` operator workflow remains the rollback path; this change does not surface it in the admin UI).
- `x.y.z` semantic-version notation for the target version (the field name `target_version` is generic enough; the wire format stays 7-char SHA-like in v1).
- Deleting `/api/sign/boot-config`. The legacy endpoint stays so existing Pis continue to upgrade. The new endpoint serves new code.
- Returning the long-form SHA on any endpoint. The long form is derivable locally via `short_sha_inverse` (or by reading `git rev-parse HEAD` on the Pi) and isn't needed on the wire for any current operation.

## Capabilities

### New Capabilities

- `pi-upgrade-settings`: New "Pi Upgrade Control" sub-section on the existing `/settings` page. Three read-only version fields (Flask, Pi, Target) and one editable Target input (empty means "track Flask"). Target version persists as `sign.target_version` on `SignSettings` (always concrete on the wire — Flask resolves operator-pin vs Flask-self). Saved through the existing `POST /api/config` endpoint — no new endpoint.
- `pi-power-commands`: New `type=command` actions (`force-upgrade`, `restart`, `shutdown`) routed through `MessageManager.dispatch` to per-action handlers on the Pi. Flask-side buttons on the Settings section publish these envelopes on click and surface success/failure via a transient status line that listens on the status topic.
- `pi-sign-settings-endpoint`: New `GET /api/sign/settings` endpoint that returns `{"target_version": "<7-char>", "timezone": "US/Pacific"}` — both fields always concrete. The Pi's loader calls this on boot; the response is parsed by the existing `lib_shared/boot_config` machinery (with the new field name).

### Modified Capabilities

- `mqtt-command-envelope` (introduced in `self-upgrading-matrix-controller`): the existing `type=command` dispatch grows three new actions — `force-upgrade`, `restart`, `shutdown` — registered **alongside** the existing `check-for-update` action. The new actions run in parallel during the transitional period; the `check-for-update` action is **not** dropped in this change. The envelope shape and route table are unchanged; only the registered handlers set grows. A follow-up change removes `check-for-update` once all deployed Pis are confirmed to be on the new code.
- `sign-config-schema` (introduced in `self-upgrading-matrix-controller`): `SignConfig.CURRENT_VERSION` STAYS at 2 (no schema bump). `sign.target_version` is added to `SignSettings` and defaults to Flask's running short SHA at construction time. No migration is needed — the new field is forward-compatible (pre-change code ignores the unknown key via `from_dict`) and backward-compatible (the field is read via the dataclass default if a pre-change Flask publishes a payload without it).

- **No new Python dependencies.** All new paths use existing stdlib (`json`, `os`, `subprocess`, `threading`) + existing paho-MQTT client + existing Flask.
- **No new MQTT topics.** `MQTT_STATUS_TOPIC` and `MQTT_TOPIC` are the only topics used.
- **One new Flask route:** `GET /api/sign/settings`. The existing `/api/sign/boot-config` route stays. The existing `POST /api/config` accepts the new `sign.target_version` field. `POST /api/sign/commands/<action>` is added for the three new commands.
- **Loader change is one fetch call + one truncation.** The loader's boot-time HTTP call swaps from `/api/sign/boot-config` to `/api/sign/settings`; the comparison becomes `short_sha(local) == target_version` (was `local == expected_sha`). One-line change in two spots. Fallthrough behavior (any failure → run current) is unchanged.
- **Pi is never bricked.** All three commands (`force-upgrade`, `restart`, `shutdown`) are routed through the loader's existing fallthrough — if anything fails along the way, the loader execs the existing `current/.../main.py` so the Pi can always recover. A failed `GET /api/sign/settings` on boot (same failure mode as today's `boot-config` failure) degrades to "no upgrade, fall through to current" (safe default).
