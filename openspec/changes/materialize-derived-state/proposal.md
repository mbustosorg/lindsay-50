## Why

PR #41 introduced a universal `on_change` event spine, but it dropped two things at once and left a third behind as dead weight:

1. **`EffectsCoordinator` lost its message-selection logic.** Before the refactor, the coordinator picked a message to display — including picking uniformly from the most recent N messages during idle so the sign always had *something* to show between real SMS arrivals. The new "JS pushes `pending_text`" model never re-pushed during idle, so the sign now shows only background between messages.
2. **`InMemoryMessages.get_messages()` computes derived state on every read.** `_enrich_messages()` runs the filter regex pass and the timezone formatter on every call, even though the inputs (`messages` + `filters` + `timezone`) only change on events. With a 100-message buffer and any future per-frame consumer, this wasted work compounds.
3. **The push path is vestigial.** The browser already has a `MessageManager` instance that's kept up-to-date by the existing envelope subscription (the same path the Pi uses). The JS `reRender` aggregator still calls `apply_config` and `request_message` against `window`, which the Python side wires to `coordinator.set_text` / `coordinator.apply_settings`. Both sides are dead — the manager is the source of truth, and the coordinator can just pull.

These look like three separate problems but share a root cause: the refactor introduced a "JS pushes" model where the right model was always "the manager holds the state, the coordinator pulls." Fixing them together restores a user-visible feature (idle rotation), removes wasted per-read computation, and clears the dead push path.

## What Changes

- **Make `message_manager: MessageManager` a required constructor argument on `EffectsCoordinator`.** No `None` default, no fallback path. The coordinator always has a manager; the manager is the single source of truth for both messages and config.
- **Add `EffectsCoordinator.get_display_message() -> str | None`.** This is the single selection algorithm. It reads the most recent `recent_count` non-suppressed messages from the manager. If the head of the list has an id the coordinator has not yet shown, return its body (fresh-message priority). Otherwise, pick uniformly at random from the list and return that body. Return `None` when the list is empty.
- **Throttle the pull to ~4 Hz on tick.** A `_last_message_pull: float` and a `_PULL_INTERVAL = 0.25` class-level constant gate the call. Tick consumes the cached pull result, not the live one. 4 Hz is 4× faster than any human perceives text change, and well below the 30+ FPS cost we're avoiding.
- **Wire the throttled pull into `tick()`.** When the pull returns a body, it becomes the next text the coordinator shows. The existing fade-out / swap / fade-in state machine keeps its shape; only the source of the "next text" changes from `self.pending_text` to the pull result.
- **Remove the push path entirely.**
  - `EffectsCoordinator.set_text(...)` is deleted. There is no `pending_text` slot; there is no `apply_text` JS-callable.
  - `EffectsCoordinator.start(startup_text)` is simplified to `start()` — no `startup_text` parameter. The coordinator's first pull returns the most recent message in the manager's buffer; the boot splash no longer needs a separate "show this body after the heart" hook. (The Pi already seeds the manager before the coordinator's first tick; the browser seeds via the existing REST fetch in `app_main.py`.)
  - `pending_text` is gone from the dataclass.
- **Remove the JS push surface entirely.**
  - `heart-message-manager/preview_main.py` no longer exposes `window.request_message` or `window.apply_config`. The manager's `on_change` listener (registered via `App.registerOnChange` in `app_main.py`) is the only event hook JS needs.
  - `heart-message-manager/static/preview/preview.js` no longer defines `reRender`, no longer calls `window.request_message(body)`, and no longer calls `window.apply_config(cfg)`. The file's job shrinks to "construct the page-local canvas and let the coordinator drive it."
- **Move `_enrich_messages()` from read-time to event-time in `MessageManager`.** `_handle_message()` triggers a per-entry re-enrich on add (cheap — one entry). `_handle_config()` triggers a full re-enrich when filter rules or timezone change. `get_messages()` becomes a thin read that returns the already-enriched `MessageView` list. No per-call filter regex, no per-call timezone formatting.
- **Update the coordinator's state through the existing `MessageManager` `on_change` callback.** The manager already exposes an `on_change` callback (used today to fan out to JS subscribers via `create_proxy(_on_change_js)`). This change uses the same callback to do both jobs: (1) apply the new config to the coordinator (pacing, effects rebuild, scroller color/speed) by calling `coordinator.apply_settings(...)`, and (2) fan out to JS subscribers. The `MessageManager` itself stays unaware of the coordinator; the callback is a closure constructed by the entrypoint with the coordinator in scope. JS no longer drives the rebind.
- **Clean up the dispatch-wrapping pattern in `heart-matrix-controller/main.py`.** The current main.py has a `_on_config_update(cfg_dict)` separate function and a `_dispatch_with_config(raw)` monkey-patch of `_message_mgr.dispatch` that intercepts config envelopes to call `_on_config_update`. This was the pre-refactor way to route config changes to the coordinator, and the new on_change pattern makes it redundant. After this change: `_on_change` (currently a no-op) is the single Python-side config-application point; `_on_config_update` and `_dispatch_with_config` are deleted; the MQTT client's `dispatch_callback` is bound to `_message_mgr.dispatch` directly (no wrapping). The `coordinator.start(_startup_text)` call also goes away — the coordinator's first pull produces the most recent message in the manager's buffer.
- **Wire the pre-existing `recent_count` config field to actually mean something.** It was retained for compat in the prior refactor but had no consumer. After this change it's the source of truth for idle-rotation size.

## Capabilities

### New Capabilities

- `derived-state-materialization`: the discipline that pre-computable per-message enrichment (`suppressed`, filter rules, `sender_name`, `display_time`) lives in `MessageManager` as a stored field on `MessageView`, recomputed only on the event that changes it, and read by consumers as a property access. The `EffectsCoordinator`'s `get_display_message()` method (fresh-message priority, then uniform recent-N sampling from the most recent `recent_count` non-suppressed messages) and the 4 Hz throttled pull from the manager on tick are part of this capability because the same store-once-read-many principle governs both.

### Modified Capabilities

_None — no prior spec-level requirements exist in `openspec/specs/`, so there are no requirement deltas to record._

- **No public-API break for `MessageManager.get_messages()`.** Same shape returned (a list of `MessageView`s); the only difference is *when* the views are enriched.
- **No new dependencies.** Pure refactor of existing event/read boundaries and removal of dead push surface.
- **Out of scope**: Pydantic validation on the wire boundaries (#43); Pi-vs-browser divergence (the same `MessageManager` class works in both — `is_browser` already gates the right behavior); anything outside `lib_shared/effects_coordinator.py`, `lib_shared/message_manager.py`, `lib_shared/messages.py`, `heart-message-manager/preview_main.py`, `heart-message-manager/static/preview/preview.js`, `heart-matrix-controller/main.py`, and their tests. PR #41's `d5ff585` cache work and `c769270` cleanup are independent and not coupled to this change.
