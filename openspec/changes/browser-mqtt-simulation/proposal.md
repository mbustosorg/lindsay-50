## Why

The previous `add-sign-preview-rendering` change proved that PyScript can run the device's unmodified effect code in the browser, but it kept the browser tied to the Flask server for message state: every preview tab polls `/api/live-messages?limit=1&suppress=true` every 3 s and the Flask process also runs a server-side `MessageManager` that subscribes to its own MQTT feed just to keep that polled endpoint fresh. The browser holds no real state, so a page reload re-seeds from a single `fetch()` and the in-tab coordinator forgets anything between requests.

Paho and Adafruit IO both expose MQTT-over-WebSocket on the same broker the device already uses, so the browser can connect to the broker directly and run its own `MessageManager` in WASM Python. The ring buffer then lives in the browser, server polling goes away, and the browser keeps state across reloads/navigation via browser-side storage. The Raspberry Pi simulation — message intake, filtering, ring buffer, preview — runs end-to-end in the page; the Flask server becomes a thin static host plus the Twilio/SQLite ingress path.

The cleanest way to share that contract is to **refactor the existing `MessageManager`** rather than create a browser-only sibling. The class's only server-specific assumption today is `requests` for the seed REST call and a module-level `get_config()` to read URLs. With those lifted — URLs as constructor params, network imports moved to lazy loaders, and an `is_browser: bool` flag selecting between the two I/O paths — the same class runs in both environments.

## What Changes

- **Refactor `lib_shared/message_manager.py`** so `MessageManager` no longer reads `config_reader` at import time, no longer imports `requests` at module top, and no longer imports `js` at module top. The constructor takes `messages_api_url`, `config_api_url`, `api_key`, and an `is_browser: bool = False` flag. The class owns both I/O implementations internally: a `requests`-based path (lazy-imported for the device) and a `js.fetch`-based path (lazy-imported for the browser), selected by the flag. The dispatch / ring-buffer / filter / config-update logic stays in one place. The Pi's `heart-matrix-controller/main.py` continues to use the refactored class; only the call site changes (it now passes the URLs, the `api_key`, and `is_browser=False`).
- Add a browser-side MQTT-over-WebSocket client (PyScript) that opens a WebSocket to the broker's WS endpoint, subscribes to the configured topic, decodes `MessageEnvelope` JSON, and feeds envelopes into the in-browser `MessageManager`
- Add browser-side persistence for the ring buffer and `SignConfig` (IndexedDB-backed) so the in-browser state survives page reloads and SPA route changes; the user does not re-seed on every visit
- Replace the preview's 3-second `/api/live-messages` polling with MQTT-driven updates: when the in-browser `MessageManager` accepts a new `message` envelope, the existing `coordinator.request_message(body)` path runs (no API change there)
- The browser subscribes via the broker's WebSocket port using the same `MQTT_USERNAME`/`MQTT_PASSWORD` the device uses; the broker handles auth the same way for WS as for native MQTT
- **Lift the connection out of `/preview` into the base template.** The WS connection, the `MessageManager`, and the IndexedDB hydrate are owned by every admin page (via `templates/base.html`'s inline bootstrap). The buffer is current regardless of which admin page the operator is on. The `/preview` page adds its `PreviewCoordinator` + canvas on top of the same in-memory state.
- **Use the device's X-API-Key auth for the browser's seed fetch, not a session cookie.** The inline `APP_CONFIG` block in the base template includes the `X-API-Key` alongside the MQTT credentials (same trust model — gated by `@login_required`, visible only to authenticated users). The browser's `MessageManager._fetch` method includes the key in the `X-API-Key` header on the `js.fetch` call, exactly like the device's `requests.get` call. This means the browser exercises the device's exact auth code path in dev — if the device's key is misconfigured, the browser surfaces it.
- **Wipe + re-seed on app start, login, and after long disconnect.** On session start, the browser clears the IndexedDB and re-fetches the canonical message + config from `/api/messages` and `/api/config` (X-API-Key auth, same as the device). The WS then keeps the buffer up-to-date. After a configurable long-disconnect threshold (default 5 minutes), the same wipe+re-seed runs on reconnect to recover from any missed envelopes.
- **Remove the Flask process's `MessageManager` instance and its MQTT subscription** — Flask no longer needs a server-side ring buffer; the only consumer of that ring buffer was the now-replaced preview polling path. The `MessageManager` class in `lib_shared/message_manager.py` stays (the device imports it directly); only the Flask's instantiation, its `seed()` thread, and the `make_mqtt_client(_message_mgr.dispatch)` wiring go away
- **Remove the `/api/live-messages`, `/api/live-messages/seed`, and `/api/live-config` routes** — these existed solely to expose the Flask's server-side ring buffer. The Pi device never calls them; it uses its own `MessageManager` instance and seeds from the REST `/api/messages` and `/api/config` endpoints (not the `live-` ones)
- **Replace the `templates/testing.html` page** — it currently POSTs test messages and polls the `/api/live-messages` and `/api/live-config` endpoints to verify the roundtrip. Since those endpoints are gone, the page is reworked: it POSTs to `/api/test-messages` and uses the browser's MQTT-WS client (subscribed to the same topic) to verify the roundtrip locally — no Flask polling involved. The page stays a useful operator tool; the test surface moves from "Flask endpoints" to "browser-internal `MessageManager.dispatch`"
- **Replace tests that hit the removed endpoints with equivalent coverage of the new browser-internal functions**: `tests/preview_poll_test.py` is deleted (it asserted the polling URL string, which no longer exists); the `/api/live-messages` and `/api/live-config` auth tests in `tests/test_auth.py` are removed; new tests cover the refactored `MessageManager` (dispatch + on_message + suppression + ring-buffer eviction + internal `_fetch` branching on `is_browser`) and the `MqttWsClient` / `MessageBufferStore` Python wrapper signatures
- No changes to the effect code in `heart-matrix-controller/patterns/`, to `ScrollerBase`, to `MatrixScroller`/`PreviewScroller`, or to the `EffectCoordinator`/`PreviewCoordinator` split

## Capabilities

### New Capabilities

- `browser-mqtt-websocket-client`: The browser opens a WebSocket to the MQTT broker's WS endpoint (Paho or Adafruit IO, same broker as the device), subscribes to the configured topic with the configured credentials, decodes `MessageEnvelope` JSON payloads, and emits them to the in-browser `MessageManager`. The connection SHALL auto-reconnect with backoff on drop and SHALL pause cleanly when the tab is hidden. The connection lives in the base template (not `/preview` only), so the buffer is current on every admin page.
- `browser-message-manager`: The existing `MessageManager` class is refactored to be config-agnostic at the class level and to own both I/O implementations internally. It SHALL take `messages_api_url`, `config_api_url`, `api_key`, and an `is_browser: bool = False` flag as constructor parameters. The class SHALL NOT read `lib_shared.config_reader` at import time; the class SHALL NOT have `import requests` or `import js` at module top (both are lazy-loaded inside the `_fetch` method). When constructed with `is_browser=True`, the class SHALL use `js.fetch` (lazy-imported) for the seed fetch; with `is_browser=False`, it SHALL use `requests` (lazy-imported) in an `asyncio.to_thread(...)` worker. The browser and the device both use this single class with the appropriate flag value; the dispatch / ring-buffer / filter / config-update logic is shared.
- `browser-message-buffer-persistence`: The in-browser `MessageManager` state — message ring buffer and `SignConfig` — persists across page reloads and SPA route changes via browser-side storage (IndexedDB). On app start and after a long disconnect, the IndexedDB is wiped and re-seeded from the REST API. On every page navigation, the IndexedDB hydrates the in-memory state before the preview requests the first frame. The store is partitioned by browser origin and profile.
- `flask-server-cleanup`: The Flask process no longer hosts a `MessageManager` instance, no longer subscribes to the broker, and no longer serves `/api/live-messages`, `/api/live-messages/seed`, or `/api/live-config`. The Twilio ingress path (`POST /api/messages`), the SQLite + S3 storage, and the `/api/messages` and `/api/config` REST endpoints (used by the Pi device's boot seed AND the browser's per-session re-seed) stay.

### Modified Capabilities

- `sign-preview-rendering`: The preview's "polling every 3 seconds" requirement is removed. The preview updates from MQTT envelopes delivered to the in-browser `MessageManager`; the `coordinator.request_message(body)` API stays unchanged. The page no longer instantiates `fetch('/api/live-messages...')` or tracks a `lastShownBody` from polled responses; the in-browser `MessageManager`'s on-message callback is the single source of new-message signals. The WS connection itself lives in the base template; the `/preview` page adds the PyScript coordinator + canvas on top of the shared in-memory state.

  - `heart-message-manager/static/mqtt_ws_client.js` — native JS MQTT-over-WebSocket shim (WebSocket → MQTT 3.1.1 CONNECT/SUBSCRIBE/PUBLISH, auto-reconnect, pause-on-hidden)

  - `heart-message-manager/static/message_buffer_store.js` — native JS IndexedDB shim (`hydrate`, `wipe`, `putMessage` with atomic trim, `putConfig`)
  - `heart-message-manager/static/message_buffer_store.py` — PyScript wrapper around the JS shim

  - `heart-message-manager/README.md` — operator note: `mosquitto -p 1883 --ws-port 9001` for local Paho with WS

- New browser-side dependencies (Pyodide / PyScript already loaded by the preview):
  - An MQTT-over-WebSocket client (a thin native-JS shim called from PyScript — no new Pyodide package, no new PyScript dependency)
  - IndexedDB access from PyScript (via a small JS shim)
- No changes to the device (`heart-matrix-controller/`) other than the `MessageManager(...)` call site, no changes to the broker config, no changes to the Twilio ingress path

- New env / config keys (read by the base template, not the server): `MQTT_WS_URL` (broker WS endpoint), `MQTT_USERNAME`, `MQTT_PASSWORD`, `MQTT_TOPIC`, `API_KEY` (the X-API-Key the device uses, rendered into the inline `APP_CONFIG` so the browser's `MessageManager._fetch` issues the authed `js.fetch` calls), `MQTT_LONG_DISCONNECT_MS` (default 300000 ms) — same names the device uses so the operator can copy values from the device's `settings.toml`. The `is_browser` flag is NOT a config key — the device's call site uses the constructor default (`is_browser=False`); the browser's call site hardcodes `is_browser=True` in PyScript
- First-load weight grows by the JS shim sizes (a few hundred lines each, ~50 KB total) — negligible compared to the existing Pyodide + Pillow + numpy footprint
- The `lib_shared/message_manager.py` class itself is refactored — it loses the `requests` and `config_reader` imports and gains constructor injection. The class, the `InMemoryMessages` ring buffer, the `seed()` method, and the dispatch contract are otherwise unchanged; the device's call site is updated to pass URLs and a transport, but its runtime behavior is identical.
