## 1. Shared status store

- [ ] 1.1 Create `lib_shared/sign_status.py` with the `LatestSignStatus` class: `threading.RLock`-guarded in-memory store with `update(snapshot_dict)` (validates required keys), `snapshot() -> dict | None` (defensive copy), and `received_at_wallclock() -> str | None` (ISO-8601 wall-clock timestamp at which the most recent update landed). The store is a thin holder for the latest snapshot — it does NOT compute state (state is browser-side per the design). Constants for the required-keys list (`REQUIRED_SNAPSHOT_KEYS = (...)`) are exported so the test suite and any future config layer can reference them by name.
- [ ] 1.2 Add `tests/test_sign_status.py` with round-trip tests: `update()` stores the dict under the lock, `snapshot()` returns a defensive copy (mutating the returned dict does not affect the store), `received_at_wallclock()` returns an ISO-8601 string after `update()` and `None` before any `update()`, `snapshot()` on an empty store returns `None`, and a payload missing a required key is rejected (raises ValueError, store unchanged). Cover the concurrent-update case (two threads call `update()` simultaneously — the lock prevents torn reads).

The existing `to_dict()` is unchanged. **Also change the existing `StatusWriter` throttle from `STATUS_WRITE_INTERVAL_S = 3.0` to `STATUS_WRITE_INTERVAL_S = 5.0`** (still well under the loader's 8s probe headroom; the 5s value is now shared with the MQTT publish path). The `tick()` method is extended to publish the MQTT envelope in the same call as the file write.
- [ ] 2.2 Create `heart-matrix-controller/status_publisher.py` with the `StatusPublisher` class: long-lived paho publisher for the status flow. Constructor: opens a single `mqtt.Client`, calls `connect_async(...)` + `loop_start()`. `publish(payload_dict) -> bool`: calls `client.publish(topic, payload.encode(), qos=0)` (thread-safe, non-blocking — the loop thread handles the network); returns True on success, False on any error or non-success `rc`. On a `publish()` returning a non-success `rc`, a `threading.Timer` schedules a reconnect attempt 5 seconds later. `close()`: calls `loop_stop()` + `disconnect()`. The existing `publish_envelope` (in `lib_shared/paho_mqtt_client.py`) keeps its fresh-client-per-call pattern — `StatusPublisher` is a separate, status-only path.
- [ ] 2.3 In `heart-matrix-controller/main.py`, instantiate `StatusPublisher` at startup (after the existing MQTT client setup) and pass it to `StatusWriter` (or call `StatusPublisher.publish()` directly from `StatusWriter.tick()`). On shutdown, call `StatusPublisher.close()` in the existing `try/finally` around the render loop. The `StatusWriter.tick()` method now does: build snapshot → call `.to_mqtt_dict()` → call `os.replace(.status.json, ...)` (existing) → call `_status_publisher.publish(payload)` (new) — all in the same `tick()` call, at the 5s throttle.
- [ ] 2.4 Add `MQTT_STATUS_TOPIC` to `REQUIRED_KEYS` in `heart-matrix-controller/main.py`. The config reader's env-override path already takes precedence over `settings.toml`, so no new logic is needed — just declare the key. The resolution helper `MQTT_STATUS_TOPIC or f"{MQTT_TOPIC}-status"` lives in `main.py` and feeds both the `StatusPublisher` constructor and the `REQUIRED_KEYS` check.
- [ ] 2.5 In `lib_shared/paho_mqtt_client.py`, extend `PahoMqttClient.__init__` with optional `status_dispatch_callback` + `status_topic` parameters (Flask-side subscriber extension; the Pi-side publisher is a separate class, see 2.2). The existing `dispatch_callback` + `topic` are unchanged. In the subscriber loop, `on_connect` calls `client.subscribe([(topic, 1), (status_topic, 1)])` when both are set; `on_message` dispatches by `msg.topic` to the right callback. The two callbacks' exception paths MUST be isolated — a raise in `status_dispatch_callback` does not affect `dispatch_callback` and vice versa. The `publish_envelope` method is unchanged.

- [ ] 4.1 In `heart-message-manager/templates/dashboard.html`, replace the hardcoded green "Live" pill with `<span id="sign-live-pill" data-state="offline" class="px-4 py-2 rounded-full bg-slate-100 text-slate-600 text-sm font-semibold flex items-center gap-2"><span class="w-2 h-2 bg-slate-400 rounded-full"></span>Offline</span>`. - [ ] 4.2 Create `heart-message-manager/static/sign_status.js` (new ES module). On DOM ready, the module: (a) fires **one** `fetch('/api/sign-status')` and stores the response in the in-memory snapshot IF `response.snapshot.updated_at` is newer than any WS-received snapshot (otherwise drop the fetch response); (b) opens a second `createMqttWsClient` instance with `{mqttWsUrl, username, password, topic: mqttStatusTopic, onEnvelope: (payload) => decodeAndStoreSnapshot(payload)}` — the `onEnvelope` callback decodes the JSON payload, validates required keys, and replaces the in-memory snapshot; (c) starts a 5-second `setInterval` that re-evaluates state and re-renders the DOM; (d) wires `setInterval` cleanup to `beforeunload`. The module exports the `LIVE_THRESHOLD_S = 15`, `UNKNOWN_THRESHOLD_S = 30`, and `stateFromAge(ageSeconds) -> "live" | "unknown" | "offline"` helpers as named exports. State mapping for the pill: `live` → green classes + pulse class + text "Live"; `unknown` → amber classes + no-pulse + text "Unknown"; `offline` → grey classes + no-pulse + text "Offline". The WS client's connection state is surfaced to a small secondary indicator next to the pill (e.g., `<span id="sign-status-ws-state">…</span>`), parallel to the existing `#mqtt-status` pattern.
- [ ] 4.3 Include `<script src="{{ url_for('static', filename='sign_status.js') }}" defer></script>` in `heart-message-manager/templates/base.html` so the script loads on every authenticated page (not just dashboard — keeps the logic DRY and matches the existing `mqtt_ws_client.js` inclusion pattern). The script MUST short-circuit gracefully (no-op) on pages that have neither `#sign-live-pill` nor `[data-sign-status-field]` (so it's safe to include globally).
