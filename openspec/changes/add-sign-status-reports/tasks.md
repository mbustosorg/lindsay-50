## 1. Shared status store

- [ ] 1.1 Create `lib_shared/sign_status.py` with the `LatestSignStatus` class: `threading.RLock`-guarded in-memory store with `update(snapshot_dict)` (validates required keys), `snapshot() -> dict | None` (defensive copy), and `received_at_wallclock() -> str | None` (ISO-8601 wall-clock timestamp at which the most recent update landed). The store is a thin holder for the latest snapshot — it does NOT compute state (state is browser-side per the design). Constants for the required-keys list (`REQUIRED_SNAPSHOT_KEYS = (...)`) are exported so the test suite and any future config layer can reference them by name.
- [ ] 1.2 Add `tests/test_sign_status.py` with round-trip tests: `update()` stores the dict under the lock, `snapshot()` returns a defensive copy (mutating the returned dict does not affect the store), `received_at_wallclock()` returns an ISO-8601 string after `update()` and `None` before any `update()`, `snapshot()` on an empty store returns `None`, and a payload missing a required key is rejected (raises ValueError, store unchanged). Cover the concurrent-update case (two threads call `update()` simultaneously — the lock prevents torn reads).

- [ ] 4.1 In `heart-message-manager/templates/dashboard.html`, replace the hardcoded green "Live" pill with `<span id="sign-live-pill" data-state="offline" class="px-4 py-2 rounded-full bg-slate-100 text-slate-600 text-sm font-semibold flex items-center gap-2"><span class="w-2 h-2 bg-slate-400 rounded-full"></span>Unknown</span>`. The new placeholder uses `data-state` so the JS can toggle the color and animation classes based on the browser-computed state.
- [ ] 4.2 Create `heart-message-manager/static/sign_status.js` (new ES module). On DOM ready, the module: (a) fires **one** `fetch('/api/sign-status')` and stores the response in the in-memory snapshot IF `response.snapshot.updated_at` is newer than any WS-received snapshot (otherwise drop the fetch response); (b) opens a second `createMqttWsClient` instance with `{mqttWsUrl, username, password, topic: mqttStatusTopic, onEnvelope: (payload) => decodeAndStoreSnapshot(payload)}` — the `onEnvelope` callback decodes the JSON payload, validates required keys, and replaces the in-memory snapshot; (c) starts a 5-second `setInterval` that re-evaluates state and re-renders the DOM; (d) wires `setInterval` cleanup to `beforeunload`. The module exports the `LIVE_THRESHOLD_S = 60`, `UNSURE_THRESHOLD_S = 120`, and `stateFromAge(ageSeconds) -> "live" | "unsure" | "offline"` helpers as named exports. State mapping for the pill: `live` → green classes + pulse class + text "Live"; `unsure` → amber classes + no-pulse + text "Live"; `offline` → grey classes + no-pulse + text "Unknown". The WS client's connection state is surfaced to a small secondary indicator next to the pill (e.g., `<span id="sign-status-ws-state">…</span>`), parallel to the existing `#mqtt-status` pattern.
- [ ] 4.3 Include `<script src="{{ url_for('static', filename='sign_status.js') }}" defer></script>` in `heart-message-manager/templates/base.html` so the script loads on every authenticated page (not just dashboard — keeps the logic DRY and matches the existing `mqtt_ws_client.js` inclusion pattern). The script MUST short-circuit gracefully (no-op) on pages that have neither `#sign-live-pill` nor `[data-sign-status-field]` (so it's safe to include globally).
