## Why

The Pi already writes a rich runtime snapshot to `heart-matrix-controller/.status.json` (PID, active SHA, started_at, updated_at, uptime_seconds, mqtt_connected, last_tick_age_ms, messages_rendered, last_error — see `heart-matrix-controller/status.py:StatusSnapshot`) every 3s via the throttled `StatusWriter`. The loader reads that file to validate staged worktrees (`heart-matrix-controller/loader.py:probe`) — it's the **only** consumer. The Flask server has **zero visibility** into the sign's runtime state: there's no MQTT topic carrying the snapshot, no Flask subscriber, no UI element that shows real sign health.

Two pieces of UI actively misrepresent reality today:

1. **Dashboard page** has a static green "Live" pill (`templates/dashboard.html:10-13`) that is hardcoded HTML — it does not reflect any runtime signal from the sign. Operators have no way to tell whether the sign is actually rendering.
2. **Settings page** has no sign-health section at all — the operator can't see the running version, last-start time, MQTT-connected flag, or last error without SSHing to the Pi and reading `.status.json` themselves.

The fix has to handle two distinct use-cases:

- **Live updates while the dashboard is open** — the operator is watching; the pill should transition `live → unsure → offline` as state changes.
- **Load-time hydration on login** — the operator opens the dashboard after being away; they should immediately see the current state, not "Unknown" until the next 30s heartbeat.

The codebase already has a strong precedent for "device → Flask/browser over MQTT" — the existing `MQTT_TOPIC` carries `MessageEnvelope`s (`type` + `payload`) and the browser subscribes to that topic via the MQTT-WS bridge (`static/mqtt_ws_client.js`'s `createMqttWsClient({...topic})`). This change extends that pattern to a second topic for status, with Flask acting as a small in-memory cache so the browser can hydrate on load.

The `StatusSnapshot` is already the right shape — schema-versioned, atomic, throttled — so this change is wiring (publish + Flask subscriber + load-time fetch + browser WS + UI), not new state design.

## What Changes

- The Pi (the device side) **publishes** a serialized `StatusSnapshot` to a new MQTT status topic on a 30s cadence (separate from the 3s `.status.json` write — `.status.json` is consumed by the loader and stays at its existing cadence; the MQTT publish is for the admin UI and is throttled to a coarser cadence to avoid broker spam).
- The **Flask server** subscribes to that status topic and keeps the latest received snapshot in an in-memory store guarded by `threading.RLock` (the same pattern `SignConfig` uses for runtime config in `lib_shared/models.py:421`). Flask surfaces the latest snapshot via `GET /api/sign-status` (always returns 200; `null` snapshot if none received yet).
- The **browser** does a single load-time `fetch('/api/sign-status')` on page load to hydrate the UI with the current state. After load, the browser subscribes to the status topic via a **second `createMqttWsClient` instance** (scoped to the status topic only) for live updates going forward. The existing envelope-flow client (subscribed to `MQTT_TOPIC`) is untouched. The load-time fetch is hydration, NOT polling — the browser does not call `fetch` on a timer.
- The browser computes the sign's state (`live` / `unsure` / `offline`) from the snapshot's age, with thresholds defined as constants in `sign_status.js` (default: `live` < 60s, `unsure` 60-120s, `offline` > 120s or never). A small local `setInterval` (5s cadence) re-evaluates state so the pill transitions even when no new message arrives. The interval is purely a UI re-render cadence — it does NOT poll the server.
- The Dashboard's hardcoded green "Live" pill becomes **dynamic**: it reads the latest snapshot (from either the load-time fetch or the WS subscription) and applies the matching CSS class. The pulse animation is preserved for the `live` state.
- The Settings page gains a new **Sign Health** section at the top (above "Sign Name") showing the latest snapshot fields (running SHA, started_at, uptime, mqtt_connected, last_error) and the timestamp the browser received the snapshot. The values are read-only — operator controls live on a different page (or aren't wired in this change). The Sign Health section hides the field slots and shows a "No status received yet" placeholder when the browser has never received a snapshot.
- `MQTT_STATUS_TOPIC` is added to both `settings.toml` files, with a documented derivation rule (the existing `MQTT_TOPIC` + `-status` suffix is the canonical default; operators can override). Environment variables take precedence. The resolved topic is exposed via the existing `window.APP_CONFIG.mqttStatusTopic` injection in `base.html` so `sign_status.js` can find it without re-parsing the URL.
- The Pi publishes via a new helper on `PahoMqttClient` (`publish_status(payload_dict, topic)`) that follows the same fresh-client-per-call pattern as the existing `publish_envelope`, but uses QoS 0 (fire-and-forget) so a slow broker can't stall the render loop. The existing `publish_envelope` is unchanged.

## Capabilities

### New Capabilities

- `sign-status-reports`: the Pi publishes a serialized `StatusSnapshot` to a new `MQTT_STATUS_TOPIC` on a 30s cadence. Flask subscribes to that topic and keeps the latest snapshot in a `threading.RLock`-guarded in-memory store (`lib_shared/sign_status.py:LatestSignStatus`). Flask exposes the latest snapshot via `GET /api/sign-status` (always returns 200; the snapshot field is `null` if none has been received since Flask started). The browser hydrates on page load with a single `fetch('/api/sign-status')`, then subscribes to the status topic via a second `createMqttWsClient` instance (independent of the existing envelope-flow client) for live updates. The browser computes the state from the snapshot's age (`live` < 60s, `unsure` 60-120s, `offline` > 120s or never) and re-evaluates every 5 seconds via a local timer so the pill transitions even when no new message arrives. The load-time fetch is hydration, NOT polling — the browser does not call `fetch` on a timer. The status publish uses QoS 0 so a flaky broker cannot stall the render loop; the loader's `.status.json` write (the existing 3s-throttled `StatusWriter`) is unchanged.

### Modified Capabilities

- *(none)* — the existing `MQTT_TOPIC` envelope publish/subscribe path on Flask is untouched. The status flow rides a separate topic and a separate browser-side client. The Flask `PahoMqttClient` is extended with a second `status_dispatch_callback` + `status_topic` pair; the existing envelope path is unchanged.

### Future Considerations (Out of Scope, Parked)

- **Events/logs topic.** A separate MQTT topic for discrete events — `rebooted`, `upgrading`, `new message received`, `new config received`, etc. — is **explicitly out of scope** for this change and is parked as a follow-up. Two reasons it lands in its own change, not as part of this one:
  1. **Different shape and cadence.** The current proposal publishes a *periodic snapshot* (state of the world every 30s). Events are *discrete appends* (something happened, once). They have different consumers and different storage assumptions — mixing them on the same topic would force every reader to disambiguate by `type` field, which is the failure mode this change deliberately avoids by giving the status flow its own topic.
  2. **Pull-on-demand, not real-time subscribe.** Adafruit IO retains feed history (last N messages), so events can be retrieved on demand via the AIO REST API when an operator opens a debug view, without Flask having to subscribe in real-time and buffer every event. The status flow, by contrast, needs a real-time subscription because the dashboard pill requires sub-minute freshness — the operator should see "the sign just went offline" within a polling interval, not after opening a debug page. This real-time-vs-pull asymmetry argues against sharing an MQTT topic or a Flask handler between the two flows.
  3. **Distinct wire shape.** Status payloads are the existing `StatusSnapshot` (scalar fields, ~9 keys). Event payloads will be a small dict (`{event: "rebooted", at: "..."}`, `{event: "upgrading", from_sha: "...", to_sha: "..."}`, etc.) — different schema, different validator, different UI surface. Trying to unify them under the existing `MessageEnvelope` shape (`type` + `payload`) would force a third `type` value into the envelope contract, which the existing envelope consumers (browser MQTT-WS bridge, Flask envelope handler) would have to learn to ignore.

  The follow-up change will add a third MQTT topic (`MQTT_EVENTS_TOPIC`, default `{MQTT_TOPIC}-events`), publish a small event dict from the Pi when notable things happen, and surface a "Recent Events" section on a debug page that pulls from the AIO REST API on page load (no Flask real-time subscriber).

## Impact

- **New files:**
  - `lib_shared/sign_status.py` — `LatestSignStatus` class: `threading.RLock`-guarded in-memory store with `update(snapshot_dict)`, `snapshot() -> dict | None` (defensive copy), `age_seconds() -> float | None`, and the monotonic-clock `received_at_monotonic` field for staleness tracking. Used by Flask (the subscriber side) and a unit test that injects synthetic snapshots and asserts `age_seconds()` and the lock semantics.
  - `heart-message-manager/static/sign_status.js` — new ES module that does a single `fetch('/api/sign-status')` on page load (load-time hydration), opens a second `createMqttWsClient` instance for `MQTT_STATUS_TOPIC` (live updates), and renders the Dashboard pill + Settings-page Sign Health section. Contains: a `stateFromAge(ageSeconds)` helper, the `LIVE_THRESHOLD_S = 60` / `UNSURE_THRESHOLD_S = 120` constants, a local `setInterval` that re-evaluates state every 5 seconds, and an MQTT-WS status handler that surfaces WS connection state to a small indicator next to the pill. The load-time fetch and the WS subscription both feed the same in-memory snapshot; a received WS message takes precedence over a stale load-time response (compare `updated_at`).
  - `tests/test_sign_status.py` — round-trip tests for `LatestSignStatus`: `update()` stores the dict under the lock, `snapshot()` returns a defensive copy (mutating the returned dict does not affect the store), `age_seconds()` returns the right number with a fake monotonic clock, `snapshot()` on an empty store returns `None`. Cover the concurrent-update case (two threads call `update()` simultaneously — the lock prevents torn reads).
  - `tests/test_sign_status_endpoint.py` — covers the Flask endpoint: 200 with `snapshot` on stored data, 200 with `snapshot: null` on empty store, content-type `application/json`.
  - `tests/test_status_to_mqtt_dict.py` — covers the `StatusSnapshot.to_mqtt_dict` serializer: drops `pid`, keeps all other fields, produces a JSON-roundtrip-safe dict.
  - `tests/test_paho_status_publish.py` — covers `PahoMqttClient.publish_status` with a mock `mqtt.Client` (assert that `publish` is called with `qos=0`, that `wait_for_publish` is NOT called, that `loop_start` + `loop_stop` are paired, and that errors return False without raising).
- **Modified files:**

The first publish happens 30s after boot (so the boot snapshot reflects a settled render loop, not the initial boot state). Returns True on successful enqueue, False on any error. The existing `publish_envelope` is unchanged.

In the subscriber loop, `on_connect` calls `client.subscribe([(topic, 1), (status_topic, 1)])` when both are set; `on_message` dispatches by `msg.topic` to the right callback.

- **No new Python dependencies.** `threading.Timer` and `threading.RLock` are stdlib.
- **No new JavaScript dependencies.** `sign_status.js` uses the existing `createMqttWsClient` shim from `static/mqtt_ws_client.js`.
- **One new Flask route:** `GET /api/sign-status` (one-shot load-time hydration, NOT polled). The browser calls it once on page load.
- **One extra WebSocket connection per browser tab.** A second `createMqttWsClient` instance adds one WS connection (to the same broker, same auth). For a single-operator admin UI, this is negligible. The two clients are independent — a status-WS reconnect storm doesn't affect the envelope-WS.
- **No `.status.json` change.** The loader's 3s-throttled file write is unchanged; the MQTT publish is a second consumer of the same `StatusSnapshot` dataclass.
- **No `MessageEnvelope` change.** Status publishes are raw dicts on a different topic; the envelope contract is untouched.
