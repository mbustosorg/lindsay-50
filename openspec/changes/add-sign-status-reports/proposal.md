## Why

The Pi already writes a rich runtime snapshot to `heart-matrix-controller/.status.json` (PID, active SHA, started_at, updated_at, uptime_seconds, mqtt_connected, last_tick_age_ms, messages_rendered, last_error — see `heart-matrix-controller/status.py:StatusSnapshot`) every 3s via the throttled `StatusWriter`. The loader reads that file to validate staged worktrees (`heart-matrix-controller/loader.py:probe`) — it's the **only** consumer. The Flask server has **zero visibility** into the sign's runtime state: there's no MQTT topic carrying the snapshot, no Flask subscriber, no UI element that shows real sign health.

Two pieces of UI actively misrepresent reality today:

1. **Dashboard page** has a static green "Live" pill (`templates/dashboard.html:10-13`) that is hardcoded HTML — it does not reflect any runtime signal from the sign. Operators have no way to tell whether the sign is actually rendering.
2. **Settings page** has no sign-health section at all — the operator can't see the running version, last-start time, MQTT-connected flag, or last error without SSHing to the Pi and reading `.status.json` themselves.

The fix has to handle two distinct use-cases:

- **Live updates while the dashboard is open** — the operator is watching; the pill should reflect both the freshness of the message stream AND the contents of the most recent snapshot. A fresh message that says `mqtt_connected: false` is not "Live" in the operator's sense — it's running but broken.
- **Load-time hydration on login** — the operator opens the dashboard after being away; they should immediately see the current state, not "Offline" until the next 5s heartbeat.

The codebase already has a strong precedent for "device → Flask/browser over MQTT" — the existing `MQTT_TOPIC` carries `MessageEnvelope`s (`type` + `payload`) and the browser subscribes to that topic via the MQTT-WS bridge (`static/mqtt_ws_client.js`'s `createMqttWsClient({...topic})`). This change extends that pattern to a second topic for status, with Flask acting as a small in-memory cache so the browser can hydrate on load.

The `StatusSnapshot` is already the right shape — schema-versioned, atomic, throttled — so this change is wiring (publish + Flask subscriber + load-time fetch + browser WS + UI + health-aware pill + loader hold update), not new state design. The snapshot's existing fields (`mqtt_connected`, `last_error`, `last_tick_age_ms`) ARE the health signals; we just have to read them.

## What Changes

- The Pi (the device side) **publishes** a serialized `StatusSnapshot` to a new MQTT status topic on a 5-second cadence. The 5s cadence is a **unified** cadence — the same `StatusWriter` that already writes `.status.json` (consumed by the loader) now also publishes to MQTT in the same `tick()` method. One throttle constant, one code path, one place to change cadence. The 5s rate is well under Adafruit IO's free-tier publish limit (30 publishes/minute → 1 every 2s; 5s gives 6x headroom) and gives the loader's updated 17s hold 3 missed writes (15s) + 2s of slack.
- The Pi uses a **long-lived paho publisher** (new `StatusPublisher` class) for the MQTT path. paho's `loop_start()` runs a background thread that handles the network; `client.publish()` is thread-safe and just enqueues into the outgoing buffer. At 5s cadence, a fresh-client-per-publish approach would burn 17,280 TCP+TLS handshakes per day per sign — a long-lived publisher is the right pattern for a regular heartbeat. The existing `publish_envelope` (which is for irregular inbound-SMS-driven publishes) keeps its fresh-client-per-call pattern.
- The loader's `BOOT_HOLD_S` is updated from 8s to 17s to match the new 5s `.status.json` write cadence. The 17s hold gives the loader the same "3 missed writes" confidence that the dashboard pill's 15s `live` window reflects — the loader, the file mtime, and the UI all read the same signal at the same scale.
- The **Flask server** subscribes to that status topic and keeps the latest received snapshot in an in-memory store guarded by `threading.RLock` (the same pattern `SignConfig` uses for runtime config in `lib_shared/models.py:421`). Flask surfaces the latest snapshot via `GET /api/sign-status` (always returns 200; `null` snapshot if none received yet).
- The **browser** does a single load-time `fetch('/api/sign-status')` on page load to hydrate the UI with the current state. After load, the browser subscribes to the status topic via a **second `createMqttWsClient` instance** (scoped to the status topic only) for live updates going forward. The existing envelope-flow client (subscribed to `MQTT_TOPIC`) is untouched. The load-time fetch is hydration, NOT polling — the browser does not call `fetch` on a timer.
- The browser computes the sign's **state** (`live` / `unknown` / `offline`) from the snapshot's age, with thresholds defined as constants in `sign_status.js` (default: `live` < 15s, `unknown` 15-30s, `offline` > 30s or never — tighter than the previous 60/120s because at 5s publish cadence, a 15s window catches three consecutive missed publishes, which is the right "something is wrong" signal). It also computes the sign's **health** (`healthy` / `degraded`) from the snapshot's contents: `degraded` if `mqtt_connected === false` OR `last_error` is non-empty OR `last_tick_age_ms >= 5000ms` (5s — the render loop is expected to tick at ~60Hz, so a 5s last-tick-age is a real "stuck" signal).
- The Dashboard's hardcoded green "Live" pill becomes **dynamic and health-aware**, rendering one of four states based on the (state, health) matrix: `live + healthy` → green pulse "Live"; `live + degraded` → amber "Degraded"; `unknown` → amber "Unknown"; `offline` → grey "Offline". The pill reflects both the freshness of the message stream AND the contents of the most recent snapshot — a fresh message that says "I'm broken" is rendered as "Degraded", not "Live". A small local `setInterval` (5s cadence) re-evaluates state AND health so the pill transitions even when no new message arrives. The interval is purely a UI re-render cadence — it does NOT poll the server.
- The Settings page gains a new **Sign Health** section at the top (above "Sign Name") showing the latest snapshot fields (running SHA, started_at, uptime, mqtt_connected, last_error) and the timestamp the browser received the snapshot. When `health=degraded`, the section surfaces a warning banner at the top of the field table naming the failing health check (e.g., "MQTT disconnected", "Last error: <message>", "Stuck render loop (<last_tick_age_ms>ms)"). The values are read-only — operator controls live on a different page (or aren't wired in this change). The Sign Health section hides the field slots and shows a "No status received yet" placeholder when the browser has never received a snapshot.
- `MQTT_STATUS_TOPIC` is added to both `settings.toml` files, with a documented derivation rule (the existing `MQTT_TOPIC` + `-status` suffix is the canonical default; operators can override). Environment variables take precedence. The resolved topic is exposed via the existing `window.APP_CONFIG.mqttStatusTopic` injection in `base.html` so `sign_status.js` can find it without re-parsing the URL.

## Capabilities

### New Capabilities

- `sign-status-reports`: the Pi publishes a serialized `StatusSnapshot` to a new `MQTT_STATUS_TOPIC` on a 5-second cadence, unified with the existing `.status.json` file write — both happen in the same `StatusWriter.tick()` call. The MQTT publish uses a long-lived paho publisher (`StatusPublisher` class) held open by the render-loop process. Flask subscribes to that topic and keeps the latest snapshot in a `threading.RLock`-guarded in-memory store (`lib_shared/sign_status.py:LatestSignStatus`). Flask exposes the latest snapshot via `GET /api/sign-status` (always returns 200; the snapshot field is `null` if none has been received since Flask started). The browser hydrates on page load with a single `fetch('/api/sign-status')`, then subscribes to the status topic via a second `createMqttWsClient` instance (independent of the existing envelope-flow client) for live updates. The browser computes the **state** from the snapshot's age (`live` < 15s, `unknown` 15-30s, `offline` > 30s or never) AND the **health** from the snapshot's contents (`degraded` if `mqtt_connected === false` OR `last_error` is non-empty OR `last_tick_age_ms >= 5000`; `healthy` otherwise), and renders the pill as one of four states: `live + healthy` → green pulse "Live"; `live + degraded` → amber "Degraded"; `unknown` → amber "Unknown"; `offline` → grey "Offline". The browser re-evaluates state AND health every 5 seconds via a local timer so the pill transitions even when no new message arrives. The load-time fetch is hydration, NOT polling — the browser does not call `fetch` on a timer. The status publish uses QoS 0 so a flaky broker cannot stall the render loop; the loader's `.status.json` write cadence moves from 3s to 5s AND the loader's `BOOT_HOLD_S` moves from 8s to 17s (3×5s + 2s slack), keeping the loader's "3 missed writes" confidence aligned with the dashboard pill's 15s `live` window.

### Modified Capabilities

- `pi-upgrade-controls` (related, not strictly this change's capability, but ships together): the loader's `BOOT_HOLD_S` is updated from `8.0` to `17.0` to match the new 5s `.status.json` write cadence. The new value gives the loader the same "3 missed writes" confidence that the dashboard pill's 15s `live` window reflects. The loader continues to use `.status.json` mtime as its sole health-check signal; this change does not add MQTT-based loader logic. The `BOOT_HOLD_S` constant's docstring is updated to explain the new derivation (3×5s + 2s slack).

### Future Considerations (Out of Scope, Parked)

- **Events/logs topic.** A separate MQTT topic for discrete events — `rebooted`, `upgrading`, `new message received`, `new config received`, etc. — is **explicitly out of scope** for this change and is parked as a follow-up. Two reasons it lands in its own change, not as part of this one:
  1. **Different shape and cadence.** The current proposal publishes a *periodic snapshot* (state of the world every 5s). Events are *discrete appends* (something happened, once). They have different consumers and different storage assumptions — mixing them on the same topic would force every reader to disambiguate by `type` field, which is the failure mode this change deliberately avoids by giving the status flow its own topic.
  2. **Pull-on-demand, not real-time subscribe.** Adafruit IO retains feed history (last N messages), so events can be retrieved on demand via the AIO REST API when an operator opens a debug view, without Flask having to subscribe in real-time and buffer every event. The status flow, by contrast, needs a real-time subscription because the dashboard pill requires sub-minute freshness — the operator should see "the sign just went offline" within a polling interval, not after opening a debug page. This real-time-vs-pull asymmetry argues against sharing an MQTT topic or a Flask handler between the two flows.
  3. **Distinct wire shape.** Status payloads are the existing `StatusSnapshot` (scalar fields, ~9 keys). Event payloads will be a small dict (`{event: "rebooted", at: "..."}`, `{event: "upgrading", from_sha: "...", to_sha: "..."}`, etc.) — different schema, different validator, different UI surface. Trying to unify them under the existing `MessageEnvelope` shape (`type` + `payload`) would force a third `type` value into the envelope contract, which the existing envelope consumers (browser MQTT-WS bridge, Flask envelope handler) would have to learn to ignore.

  The follow-up change will add a third MQTT topic (`MQTT_EVENTS_TOPIC`, default `{MQTT_TOPIC}-events`), publish a small event dict from the Pi when notable things happen, and surface a "Recent Events" section on a debug page that pulls from the AIO REST API on page load (no Flask real-time subscriber).

## Impact

- **New files:**
  - `lib_shared/sign_status.py` — `LatestSignStatus` class: `threading.RLock`-guarded in-memory store with `update(snapshot_dict)`, `snapshot() -> dict | None` (defensive copy), and `received_at_wallclock() -> str | None` (ISO-8601 timestamp at which the most recent update landed). The store is a thin holder for the latest snapshot — it does NOT compute state (state is browser-side per the design). Used by Flask (the subscriber side) and a unit test that injects synthetic snapshots and asserts the lock and defensive-copy semantics.
  - `heart-matrix-controller/status_publisher.py` — `StatusPublisher` class: long-lived paho publisher. A reconnect attempt runs on a 5-second `threading.Timer` if `publish()` ever returns `rc != paho.MQTT_ERR_SUCCESS`.
